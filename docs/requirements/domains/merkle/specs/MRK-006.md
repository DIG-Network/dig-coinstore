# MRK-006: Leaf Hash Function

> **Authoritative requirement:** [MRK-006](../NORMATIVE.md#MRK-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

The leaf hash function defines how a `CoinRecord` is converted into the 32-byte leaf value stored in the sparse Merkle tree. The function MUST be `coin_record_hash(record) = sha256(bincode::serialize(record))`. This hash MUST be deterministic -- the same `CoinRecord` always produces the same hash regardless of platform, endianness, or serialization library version. This hash is used as the leaf value in the sparse Merkle tree (MRK-001), and any inconsistency in the hash function would cause Merkle root divergence between nodes.

---

## Specification

### Function Signature

```rust
/// Compute the leaf hash for a CoinRecord to be inserted into the sparse Merkle tree.
/// The hash MUST be deterministic: the same CoinRecord always produces the same hash.
pub fn coin_record_hash(record: &CoinRecord) -> Bytes32 {
    let serialized = bincode::encode_to_vec(record, config::standard()
        .with_fixed_int_encoding()
        .with_big_endian())
        .expect("CoinRecord serialization must not fail");
    sha256(&serialized)
}
```

### Serialization Configuration

The bincode serialization MUST use the same configuration as STO-008:

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Integer encoding | Fixed-width | Deterministic size regardless of value magnitude |
| Byte order | Big-endian | Consistent across platforms |

### Determinism Requirements

1. The same `CoinRecord` value MUST always produce the same leaf hash, regardless of:
   - CPU architecture (x86, ARM, etc.)
   - Operating system (Linux, macOS, Windows)
   - Rust compiler version (within the same bincode major version)
   - Order of struct field initialization
2. The bincode crate version MUST be pinned in `Cargo.lock` to prevent silent serialization format changes.
3. `CoinRecord` MUST derive `bincode::Encode` and `bincode::Decode` with stable field ordering.

### Usage in the Merkle Tree

- When a coin is **created** (block application), `coin_record_hash(record)` is computed and passed to `SparseMerkleTree::batch_insert` as the leaf value.
- When a coin is **spent** (spend processing), `coin_record_hash(updated_record)` is computed (with `spent_height` now set) and passed to `SparseMerkleTree::batch_update` as the new leaf value.
- When a coin is **rolled back** (rollback processing), the coin's key is passed to `SparseMerkleTree::batch_remove`.

### Hash Function

The hash function is SHA-256, consistent with all other hashing in the system (Merkle internal nodes, coin IDs, etc.).

## Acceptance Criteria

- [ ] `coin_record_hash()` function exists and returns `Bytes32`.
- [ ] Hash is computed as `sha256(bincode::serialize(record))`.
- [ ] Bincode uses fixed-width integer encoding and big-endian byte order.
- [ ] The same `CoinRecord` always produces the same hash (determinism).
- [ ] Hash changes when any field of `CoinRecord` changes (preimage sensitivity).
- [ ] The hash is used as the leaf value in `SparseMerkleTree::batch_insert` and `batch_update`.
- [ ] Bincode version is pinned in `Cargo.lock`.

---

## Implementation Notes

- **Primary file:** `src/merkle/leaf_hash.rs` or `src/merkle/mod.rs`
- **Dependencies:** STO-008 (bincode configuration), MRK-001 (leaf value consumer)
- **Edge cases:**
  - `CoinRecord` with `spent_height = 0` (unspent) vs `spent_height > 0` (spent) MUST produce different hashes.
  - `CoinRecord` with `coinbase = true` vs `coinbase = false` MUST produce different hashes even if all other fields match.
  - Two `CoinRecord` values that differ only in `timestamp` MUST produce different hashes.
- **Performance:** SHA-256 of a small bincode blob (~100 bytes) is sub-microsecond. This is not a bottleneck.
