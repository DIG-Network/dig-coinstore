# MRK-003: Persistent Internal Nodes

> **Authoritative requirement:** [MRK-003](../NORMATIVE.md#MRK-003)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

The sparse Merkle tree's internal nodes must survive process restarts. Rather than rebuilding the entire tree from coin records on every startup, internal node hashes are persisted in a dedicated RocksDB column family (`merkle_nodes`). During batch operations, only the nodes whose values change (dirty nodes) are tracked. When the block's coin record changes are flushed via `WriteBatch`, the dirty Merkle nodes are included in the same atomic write. On startup, only the root hash needs to be loaded -- child nodes are fetched on demand from the column family.

This design is specified in SPEC.md Section 13.4 (Persistent Merkle Tree).

---

## Specification

### Column Family

| CF Name | Key | Value | Description |
|---------|-----|-------|-------------|
| `merkle_nodes` | `Bytes32` (node hash or path-based key) | `Bytes32 \|\| Bytes32` (left child hash, right child hash) | Internal node storage |

### Dirty Node Tracking

```rust
impl SparseMerkleTree {
    /// Returns all nodes modified since the last flush.
    fn dirty_nodes(&self) -> &HashMap<NodeKey, (Bytes32, Bytes32)>;

    /// Clears the dirty set after a successful flush.
    fn clear_dirty(&mut self);
}
```

### Flush Integration

```rust
impl SparseMerkleTree {
    /// Appends all dirty node put/delete operations to the provided WriteBatch.
    /// This MUST be called as part of the same WriteBatch that persists coin records.
    pub fn flush_to_batch(&mut self, batch: &mut WriteBatch) -> Result<(), MerkleError>;
}
```

### Startup

```rust
impl SparseMerkleTree {
    /// Loads only the root hash from persistent storage.
    /// Child nodes are fetched lazily from merkle_nodes CF as needed.
    pub fn load_from_store(db: &DB) -> Result<Self, MerkleError>;
}
```

### Behavior

1. During `batch_insert`, `batch_update`, and `batch_remove` (MRK-001), all internal nodes whose hashes change MUST be added to the dirty set.
2. `flush_to_batch` MUST write all dirty nodes to the `merkle_nodes` CF within the provided `WriteBatch`. Nodes that become empty subtrees SHOULD be deleted rather than stored (space optimization).
3. `flush_to_batch` MUST clear the dirty set after appending operations to the batch.
4. The `WriteBatch` containing Merkle node updates MUST be the same `WriteBatch` used for coin record updates, ensuring atomicity.
5. `load_from_store` MUST read only the root hash (a single key lookup or metadata entry). It MUST NOT scan or load the entire `merkle_nodes` CF.
6. After `load_from_store`, internal nodes MUST be fetched lazily from the `merkle_nodes` CF as tree traversal requires them.

## Acceptance Criteria

- [ ] A `merkle_nodes` column family exists in the RocksDB schema.
- [ ] Dirty nodes are tracked during `batch_insert`, `batch_update`, and `batch_remove`.
- [ ] `flush_to_batch` appends dirty node writes to the provided `WriteBatch`.
- [ ] Merkle node writes and coin record writes occur in the same atomic `WriteBatch`.
- [ ] `load_from_store` reads only the root hash on startup.
- [ ] After restart, `root()` returns the same hash as before shutdown.
- [ ] After restart, proof generation (MRK-004) works correctly via lazy node loading.
- [ ] Empty subtree nodes are not persisted (space optimization).

## Implementation Notes

- **Primary file:** `src/merkle/persistent.rs`, `src/storage/schema.rs` (CF definition)
- **Dependencies:** MRK-001 (tree operations produce dirty nodes), MRK-002 (empty hash detection for deletion), STO-001 (RocksDB WriteBatch)
- **Edge cases:**
  - Crash between `root()` and `flush_to_batch`: dirty nodes not yet persisted; tree must be rebuilt from coin records on next startup if root hash was not persisted.
  - Very deep dirty paths (256 levels) produce up to 256 dirty nodes per leaf change, but shared paths across batch entries reduce total dirty count.
  - Rollback (RBK-002) deletes Merkle entries; those deletions must also go through `flush_to_batch`.
- **Storage format:** The root hash itself should be stored in a well-known metadata key (e.g., `b"merkle_root"`) in a metadata CF or the `merkle_nodes` CF.
