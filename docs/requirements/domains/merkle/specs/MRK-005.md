## Summary

`verify(proof, expected_root) -> bool` is a static method that determines whether a `SparseMerkleProof` is valid against a given trusted state root. It reconstructs the root hash by hashing the leaf value with each sibling along the 256-level path and comparing the result to the expected root. The method works identically for inclusion proofs (leaf contains a coin hash) and non-inclusion proofs (leaf contains the empty hash). Because it is static, verification requires no access to the tree's internal state, enabling lightweight clients and external validators to confirm coin membership.

## Specification

### Method

```rust
impl SparseMerkleProof {
    /// Verify this proof against the expected Merkle root.
    /// Returns true if the proof is valid, false otherwise.
    /// This is a static/standalone method -- no tree access required.
    pub fn verify(&self, expected_root: &Bytes32) -> bool;
}
```

### Algorithm

```
current_hash = hash_leaf(proof.leaf_value)
for i in 0..256:
    bit = proof.key.bit(255 - i)   // bit at level (256 - i)
    if bit == 0:
        current_hash = hash_node(current_hash, proof.siblings[i])
    else:
        current_hash = hash_node(proof.siblings[i], current_hash)
return current_hash == expected_root
```

Where:
- `proof.key.bit(n)` returns the n-th bit of the coin ID (0 = most significant).
- `hash_leaf` hashes a leaf value (may be identity if leaf values are already hashes).
- `hash_node(left, right)` = `SHA-256(left || right)`.
- `siblings[0]` is the sibling at the deepest level (closest to leaf), `siblings[255]` is the sibling at the shallowest level (closest to root).

### Behavior

1. Reconstruct the path from leaf to root using the proof's sibling hashes and the key's bit-path.
2. At each level, the key's bit determines whether the current hash is the left or right child.
3. Compare the final computed hash to `expected_root`.
4. Return `true` if they match, `false` otherwise.
5. For inclusion proofs: `proof.leaf_value` is the coin record hash. A valid proof confirms the coin exists in the committed state.
6. For non-inclusion proofs: `proof.leaf_value` is the empty leaf hash. A valid proof confirms the coin does NOT exist in the committed state.
7. The method MUST NOT access any tree state, database, or global mutable state. It operates purely on the proof data and the expected root.

## Acceptance Criteria

- [ ] `verify` is callable without a `SparseMerkleTree` instance (static/standalone).
- [ ] Returns `true` for a valid inclusion proof against the correct root.
- [ ] Returns `true` for a valid non-inclusion proof against the correct root.
- [ ] Returns `false` for a proof with a tampered sibling hash.
- [ ] Returns `false` for a proof with a tampered leaf value.
- [ ] Returns `false` for a valid proof verified against a wrong root.
- [ ] Works against any trusted root (not coupled to a specific tree instance).
- [ ] No tree state, database, or I/O accessed during verification.

## Implementation Notes

- **Primary file:** `src/merkle/proof.rs`
- **Dependencies:** None beyond the hash function (SHA-256). This is intentionally self-contained.
- **Edge cases:**
  - Proof with all-zero siblings and empty leaf value: should produce the empty tree root (verifiable against MRK-002 level-0 hash).
  - Bit ordering: the most significant bit of the key selects at the shallowest tree level. The implementation must be consistent with MRK-004's path traversal.
  - Hash function must match exactly between proof generation and verification (same leaf hashing, same node hashing).
- **Portability:** Because `verify` is static and self-contained, it can be extracted into a standalone library for light clients, wallets, or external verifiers.
