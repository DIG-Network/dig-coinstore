## Summary

Phase 1 removal validation (BLK-005) checks that every coin in the block's `removals` list exists and is currently unspent. Since the in-memory unspent set (PRF-001) is read-only during Phase 1 (no mutations have occurred yet), these checks can be parallelized using `rayon`'s `par_iter` without any locking.

For blocks with large removal lists (hundreds or thousands of coins), this parallelization provides a measurable speedup on multi-core systems. The validation is embarrassingly parallel: each removal check is independent and reads only from the immutable unspent set.

---

## Specification

### Parallel Validation Pattern

```rust
use rayon::prelude::*;

fn validate_removals(
    removals: &[CoinId],
    unspent_set: &HashSet<CoinId>,
) -> Result<(), CoinStoreError> {
    // par_iter: each check runs on a rayon worker thread
    removals.par_iter().try_for_each(|coin_id| {
        if !unspent_set.contains(coin_id) {
            // Determine whether coin is missing entirely or double-spent
            // (may require a storage lookup for the full error variant)
            return Err(CoinStoreError::CoinNotFound { coin_id: *coin_id });
        }
        Ok(())
    })
}
```

### Lock-Free Guarantee

The unspent set (`HashSet<CoinId>`) is immutable during Phase 1:
- Phase 1 is validation-only; no mutations occur.
- The exclusive write lock is already held by `apply_block`, so no other writer can modify the set.
- `par_iter` readers share an immutable reference to the set.

No `Mutex`, `RwLock`, or atomic operations are needed for the parallel reads.

### Error Handling

If any removal fails validation, `par_iter().try_for_each()` short-circuits and returns the first error. The specific error variant (`CoinNotFound` or `DoubleSpend`) depends on whether the coin exists in storage at all or exists but is already spent.

### Fallback

If `rayon` is not available or the removal list is small (e.g., fewer than 100 items), a sequential `iter().try_for_each()` is acceptable and produces identical results.

## Acceptance Criteria

- [ ] Removal validation uses `rayon` `par_iter` for parallel execution.
- [ ] Validation is lock-free (no synchronization primitives during parallel reads).
- [ ] Results match sequential validation (same errors for same inputs).
- [ ] Short-circuits on first error.
- [ ] Unspent set is not mutated during validation.
- [ ] Performance improvement measurable on multi-core systems with large removal lists.

## Implementation Notes

- **Primary file:** `src/coin_store.rs` or `src/block_application.rs`
- **Dependencies:** `rayon` crate, PRF-001 (in-memory unspent set), BLK-005 (removal validation requirement)
- **Edge cases:**
  - Empty removals list: `par_iter` on empty slice is a no-op.
  - Single removal: parallelization has negligible benefit but no correctness issue.
  - All removals invalid: `try_for_each` returns the first error encountered (nondeterministic which one if multiple fail).
  - `DoubleSpend` vs `CoinNotFound` distinction: may require a storage lookup if the coin is not in the unspent set. This lookup can also be parallelized.
