## Summary

The coinstate uses a reader-writer lock to allow maximum read concurrency while ensuring exclusive access during mutations. All query methods acquire a shared read lock, allowing unlimited concurrent readers. `apply_block` and `rollback` acquire an exclusive write lock, blocking all other access during the mutation.

`parking_lot::RwLock` is preferred over `std::sync::RwLock` for its superior performance characteristics: no poisoning, faster uncontended acquisition, and fairness guarantees that prevent writer starvation.

---

## Specification

### Lock Structure

```rust
use parking_lot::RwLock;

pub struct CoinStore {
    inner: RwLock<CoinStoreInner>,
}

struct CoinStoreInner {
    // storage backend, unspent set, cache, counters, etc.
}
```

### Lock Acquisition Rules

| Method Category | Lock Type | Examples |
|-----------------|-----------|---------|
| Queries | `inner.read()` (shared) | `get_coin_record()`, `is_unspent()`, `get_coins_by_puzzle_hash()`, `height()`, `tip_hash()` |
| Mutations | `inner.write()` (exclusive) | `apply_block()`, `rollback()` |
| Metadata reads | `inner.read()` (shared) | `unspent_count()`, `spent_count()`, `total_value()` |

### Concurrency Semantics

- **Multiple readers**: Any number of threads can hold `read()` simultaneously. They do not block each other.
- **Single writer**: Only one thread can hold `write()` at a time. All readers and other writers are blocked.
- **Writer waits for readers**: A `write()` call blocks until all current `read()` guards are dropped.
- **Readers wait for writer**: A `read()` call blocks if a `write()` guard is held.

### Why parking_lot

| Feature | `std::sync::RwLock` | `parking_lot::RwLock` |
|---------|---------------------|----------------------|
| Poisoning | Yes (panics propagate) | No |
| Uncontended perf | Moderate | ~2x faster |
| Writer starvation | Possible | Fair scheduling |
| `try_read()` / `try_write()` | Yes | Yes, plus `try_read_for()` / `try_write_for()` |

## Acceptance Criteria

- [ ] `CoinStore` wraps its inner state in `parking_lot::RwLock`.
- [ ] All query methods acquire a shared read lock.
- [ ] `apply_block` acquires an exclusive write lock.
- [ ] `rollback` acquires an exclusive write lock.
- [ ] Concurrent readers do not block each other.
- [ ] Writer blocks all readers and other writers.
- [ ] No deadlocks under normal usage patterns.

## Implementation Notes

- **Primary file:** `src/coin_store.rs`
- **Dependencies:** `parking_lot` crate, CON-001 (Send + Sync requirement)
- **Edge cases:**
  - Re-entrant locking: `parking_lot::RwLock` is not re-entrant. A method holding a read lock must not call another method that acquires a write lock (or vice versa). All lock acquisition must happen at the public API boundary.
  - Lock scope: guards must be dropped before performing I/O or other blocking operations to avoid holding the lock unnecessarily.
  - `try_write()` with timeout: may be useful for graceful degradation under high contention, but not required for initial implementation.
