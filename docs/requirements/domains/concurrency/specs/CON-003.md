# CON-003: MVCC Reads During Writes

## Summary

While `apply_block` Phase 2 is executing (mutating coin records, updating indices, computing Merkle root), concurrent readers should still be able to query the coinstate and see a consistent view of the pre-block state. This is achieved through MVCC (Multi-Version Concurrency Control) semantics.

For RocksDB, a snapshot is taken before the write batch begins. Readers that arrive during Phase 2 use this snapshot, seeing the pre-block state. For LMDB, MVCC is a natural property of its copy-on-write B-tree design -- readers always see the state as of the last committed transaction.

After the write batch commits, the chain tip is swapped atomically, and new readers see the post-block state.

---

## Specification

### RocksDB MVCC Strategy

```
Timeline:
  t0: apply_block() acquires write lock
  t1: RocksDB snapshot taken (pre-block state)
  t2: Phase 2 mutations begin (WriteBatch accumulated)
  t3: Reader arrives -> uses snapshot from t1 (sees pre-block state)
  t4: WriteBatch committed atomically
  t5: Chain tip updated
  t6: Write lock released
  t7: Reader arrives -> sees post-block state
```

### Implementation Pattern (RocksDB)

```rust
fn apply_block(&self, block: BlockData) -> Result<ApplyBlockResult, CoinStoreError> {
    let mut inner = self.inner.write(); // exclusive lock

    // Take snapshot for concurrent readers
    let snapshot = inner.db.snapshot();
    inner.read_snapshot = Some(snapshot);

    // Phase 1: validation (read-only, uses snapshot)
    // Phase 2: mutation (WriteBatch)
    let mut batch = WriteBatch::default();
    // ... accumulate mutations ...
    inner.db.write(batch)?;

    // Swap tip atomically
    inner.current_height = block.height;
    inner.tip_hash = block.block_hash;

    // Clear snapshot (new readers use live DB)
    inner.read_snapshot = None;

    Ok(result)
}
```

### LMDB MVCC Strategy

LMDB provides natural MVCC: read transactions see a consistent snapshot as of their start time. No explicit snapshot management is needed. Writers create a new version of affected pages; readers continue seeing the old pages until their transaction ends.

### Atomic Tip Swap

The chain tip (height + block hash) MUST be updated atomically after the write batch commits. This ensures that:
- Readers during Phase 2 see `height = N` (pre-block).
- Readers after Phase 2 see `height = N + 1` (post-block).
- No reader ever sees `height = N + 1` with pre-block coin state (torn read).

---

## Acceptance Criteria

- [ ] Readers during `apply_block` Phase 2 see consistent pre-block state.
- [ ] Readers after Phase 2 commit see post-block state.
- [ ] No torn reads (height advanced but coins not yet updated, or vice versa).
- [ ] RocksDB: snapshot taken before write batch, used by concurrent readers.
- [ ] LMDB: natural MVCC leveraged (no explicit snapshot needed).
- [ ] Tip swap (height + block hash) is atomic with respect to readers.

## Implementation Notes

- **Primary file:** `src/coin_store.rs`
- **Dependencies:** CON-002 (RwLock provides the exclusive write lock), storage backend (snapshot/MVCC support)
- **Edge cases:**
  - Reader that started before Phase 2 and finishes after Phase 2: must still see pre-block state (snapshot is held by the reader's guard).
  - Snapshot lifetime: the RocksDB snapshot must not be dropped while any reader is using it. This requires careful lifetime management (e.g., `Arc<Snapshot>`).
  - Rollback: same MVCC semantics apply -- readers during rollback see pre-rollback state.
  - Nested apply_block: not supported (exclusive write lock prevents it).
