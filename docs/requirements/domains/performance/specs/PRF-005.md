# PRF-005: Tiered Spent Coin Archival

> **Authoritative requirement:** [PRF-005](../NORMATIVE.md#PRF-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 13.1

## Summary

As the chain grows, the number of spent coins vastly exceeds unspent coins. Spent coins older than the rollback window are immutable (they can never be rolled back) and are rarely queried. Tiered archival moves these cold records to a separate `archive_coin_records` column family, reducing the size of the hot tier and improving query performance for active coins.

Archival is a background process that does not block `apply_block` or queries. It is controlled by `ArchiveConfig`, which specifies when coins become eligible for archival and whether archived coins should be pruned from the hot tier.

---

## Specification

### Configuration

```rust
struct ArchiveConfig {
    /// Coins spent more than this many blocks ago are eligible for archival.
    /// Default: DEFAULT_ROLLBACK_WINDOW (1,000)
    archive_after_blocks: u64,

    /// If true, archived coins are deleted from the hot tier.
    /// If false, they remain in both tiers (archive is a copy).
    /// Default: false
    prune_archived: bool,
}
```

### Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `DEFAULT_ROLLBACK_WINDOW` | 1,000 | Blocks before a spent coin is eligible for archival |

### Column Families

| CF | Contents |
|----|----------|
| `coin_records` (hot) | All non-archived coin records, full indices |
| `archive_coin_records` (cold) | Archived spent coin records |

### Archival Process

1. A background task periodically scans for spent coins where `current_height - spent_height > archive_after_blocks`.
2. Eligible coins are copied to `archive_coin_records`.
3. If `prune_archived` is true, the coin is deleted from `coin_records` and all hot-tier indices.
4. If `prune_archived` is false, the coin remains in both tiers.

### Query Behavior

- Queries against the hot tier are unaffected.
- If a coin is not found in the hot tier and `prune_archived` is true, the query SHOULD fall through to `archive_coin_records`.
- The hot tier MUST retain full indices for all non-archived coins.

---

## Acceptance Criteria

- [ ] `ArchiveConfig` struct with `archive_after_blocks` and `prune_archived` fields.
- [ ] Background process migrates eligible spent coins to `archive_coin_records`.
- [ ] `prune_archived = false` keeps coins in both tiers.
- [ ] `prune_archived = true` removes coins from hot tier after archival.
- [ ] Hot tier retains full indices for non-archived coins.
- [ ] Archival does not block `apply_block` or queries.
- [ ] Archived coins are queryable via fallback to archive CF when pruned.

---

## Implementation Notes

- **Primary file:** `src/archive.rs` or `src/storage/archive.rs`
- **Dependencies:** Storage layer (column family management), CON-002 (archival must respect the locking strategy)
- **Edge cases:**
  - Chain reorg beyond `archive_after_blocks`: should not happen if `archive_after_blocks >= DEFAULT_ROLLBACK_WINDOW`, but if it does, archived coins cannot be unarchived. This is a configuration error.
  - `archive_after_blocks = 0`: all spent coins are immediately eligible. Not recommended.
  - Archival of coins spent in the same block they were created: eligible after `archive_after_blocks`.

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| `test_archive_eligibility` | Unit | Coin spent 1001 blocks ago with threshold 1000 | Eligible for archival |
| `test_archive_not_eligible` | Unit | Coin spent 999 blocks ago with threshold 1000 | Not eligible |
