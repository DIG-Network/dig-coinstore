## Summary

The crate MUST include `criterion` benchmark tests that measure and verify the performance targets defined in SPEC Section 13.12. These benchmarks serve as regression guards ensuring that optimization invariants are preserved across code changes. Each benchmark corresponds to a specific operation and its latency/throughput ceiling.

---

## Specification

### Performance Targets

The following 11 targets MUST have corresponding `criterion` benchmarks:

| # | Operation | Target |
|---|-----------|--------|
| 1 | `is_unspent(coin_id)` | < 1 us |
| 2 | `get_coin_record(coin_id)` cache hit | < 5 us |
| 3 | `get_coin_record(coin_id)` cache miss | < 100 us |
| 4 | `get_coin_records_by_puzzle_hash(unspent)` | < 1 ms for 100 coins |
| 5 | `apply_block(1000 additions, 500 removals)` | < 50 ms |
| 6 | `rollback_to_block(1 block)` | < 100 ms |
| 7 | `num_unspent()` / `total_unspent_value()` | < 1 us |
| 8 | `state_root()` | < 1 us |
| 9 | `get_coin_proof(coin_id)` | < 10 ms |
| 10 | Startup (10M coins) | < 5 s |
| 11 | Snapshot generation (10M unspent) | < 30 s |

### Benchmark Framework

- All benchmarks MUST use the `criterion` crate.
- Benchmarks MUST reside under `benches/` in the crate root.
- Each benchmark MUST be runnable via `cargo bench`.
- Benchmarks SHOULD use realistic data sizes (not toy inputs) for meaningful measurements.

### Measurement Requirements

- Latency benchmarks (targets 1-9) MUST use `criterion::Criterion::bench_function` or `bench_with_input`.
- Large-scale benchmarks (targets 10-11) MUST use a pre-populated data set of 10M coin records.
- All benchmarks MUST report median and standard deviation.
- Benchmarks SHOULD be annotated with the target threshold for human review.

---

## Acceptance Criteria

- [ ] `criterion` is listed as a `[dev-dependencies]` in `Cargo.toml`.
- [ ] All 11 performance targets have corresponding benchmark functions.
- [ ] Benchmarks are runnable via `cargo bench`.
- [ ] Each benchmark measures the operation specified in the target table.
- [ ] Benchmark results include median latency and standard deviation.
- [ ] Large-scale benchmarks (startup, snapshot) use 10M coin data sets.

---

## Implementation Notes

- **Primary file:** `benches/performance.rs` or split into `benches/prf_*.rs`
- **Dependencies:** `criterion` crate, test fixtures for generating coin records at scale
- **Edge cases:**
  - CI environments may not meet the absolute targets due to slower hardware; benchmarks SHOULD be used for regression detection rather than absolute pass/fail in CI.
  - 10M coin benchmarks require significant memory (~400 MB for the unspent set alone); these MAY be gated behind a `--features bench-large` flag.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| `bench_is_unspent` | Benchmark | Measure `is_unspent(coin_id)` latency | < 1 us median |
| `bench_get_coin_record_cache_hit` | Benchmark | Measure `get_coin_record` with warm cache | < 5 us median |
| `bench_get_coin_record_cache_miss` | Benchmark | Measure `get_coin_record` with cold cache | < 100 us median |
| `bench_get_records_by_puzzle_hash` | Benchmark | Query 100 unspent coins by puzzle hash | < 1 ms median |
| `bench_apply_block` | Benchmark | Apply block with 1000 additions, 500 removals | < 50 ms median |
| `bench_rollback_one_block` | Benchmark | Rollback a single block | < 100 ms median |
| `bench_num_unspent` | Benchmark | Read `num_unspent()` counter | < 1 us median |
| `bench_total_unspent_value` | Benchmark | Read `total_unspent_value()` counter | < 1 us median |
| `bench_state_root` | Benchmark | Read `state_root()` | < 1 us median |
| `bench_get_coin_proof` | Benchmark | Generate a coin inclusion proof | < 10 ms median |
| `bench_startup_10m` | Benchmark | Startup with 10M coins in storage | < 5 s |
| `bench_snapshot_10m` | Benchmark | Generate snapshot with 10M unspent coins | < 30 s |
