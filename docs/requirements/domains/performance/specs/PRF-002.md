# PRF-002: LRU Coin Record Cache

> **Authoritative requirement:** [PRF-002](../NORMATIVE.md#PRF-002)

> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

An LRU (Least Recently Used) cache sits between the public query API and the storage layer, caching full `CoinRecord` structs by `CoinId`. This avoids redundant storage reads for frequently-accessed coins (e.g., coins queried during mempool validation, then again during block application).

The cache uses write-through semantics: when `apply_block` creates or updates a coin record, the cache is updated simultaneously. On rollback, the entire cache is invalidated rather than attempting selective invalidation, because rollback can affect an arbitrary number of records and selective invalidation risks inconsistency.

The cache is not persisted to disk. On restart, it starts cold and warms up organically through query traffic.

---

## Specification

### Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `DEFAULT_COIN_CACHE_CAPACITY` | 1,000,000 | Maximum number of entries in the LRU cache |

### Data Structure

```rust
use lru::LruCache;

struct CoinStoreInner {
    coin_cache: LruCache<CoinId, CoinRecord>,
    // ...
}
```

### Semantics

| Operation | Behavior |
|-----------|----------|
| Query (cache hit) | Return cached `CoinRecord` without touching storage |
| Query (cache miss) | Read from storage, insert into cache, return |
| `apply_block` (addition) | Insert new `CoinRecord` into cache (write-through) |
| `apply_block` (removal) | Update cached record with `spent_height` (write-through) |
| Rollback | Invalidate (clear) the entire cache |
| Startup | Cache starts empty (not persisted) |

### Eviction

When the cache reaches `DEFAULT_COIN_CACHE_CAPACITY`, the least recently used entry is evicted. No write-back is needed because all writes go through to storage immediately (write-through).

---

## Acceptance Criteria

- [ ] Cache hit returns `CoinRecord` without storage access.
- [ ] Cache miss falls through to storage and populates the cache.
- [ ] `apply_block` additions are written through to the cache.
- [ ] `apply_block` removals update the cached record's `spent_height`.
- [ ] Rollback clears the entire cache.
- [ ] Cache capacity is bounded at `DEFAULT_COIN_CACHE_CAPACITY` (1M).
- [ ] Cache is not persisted; starts empty on restart.
- [ ] LRU eviction occurs when capacity is exceeded.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` or `src/cache.rs`
- **Dependencies:** Storage layer (for cache miss fallback), BLK-007/BLK-008 (write-through during apply_block)
- **Edge cases:**
  - Querying a coin that was just evicted: cache miss, re-read from storage.
  - Rollback invalidation is O(1) amortized (just clear the map).
  - Concurrent readers may observe stale cache entries if a write is in progress; mitigated by the RwLock strategy (CON-002).
