# PRF-006: Snapshot-Based Fast Sync

> **Authoritative requirement:** [PRF-006](../NORMATIVE.md#PRF-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

New nodes joining the network should not need to replay the entire chain history to build their coinstate. Snapshot-based fast sync allows a node to download a checkpoint snapshot of the coinstate at a known height, verify its integrity via a Merkle root against a trusted block header, restore the state, and resume normal block application from the snapshot height.

This reduces initial sync time from hours (replaying millions of blocks) to minutes (downloading and restoring a snapshot). The Merkle root embedded in the snapshot provides cryptographic assurance that the snapshot represents a valid coinstate at the claimed height.

---

## Specification

### Snapshot Contents

A snapshot MUST contain:
1. All coin records (unspent and recently-spent) at the snapshot height.
2. The Merkle tree state (all leaves and intermediate nodes).
3. The chain tip metadata (height, block hash, timestamp).
4. Aggregate counters (unspent_count, spent_count, total_value).
5. A Merkle root covering the entire snapshot.

### Snapshot Lifecycle

| Step | Actor | Description |
|------|-------|-------------|
| Create | Existing node | Serialize coinstate at a checkpoint height into a snapshot file |
| Distribute | Network | Make snapshot available for download (out of scope for coinstate crate) |
| Download | New node | Fetch snapshot from a peer or CDN (out of scope for coinstate crate) |
| Verify | New node | Compute Merkle root of snapshot, compare against trusted header |
| Restore | New node | Deserialize snapshot into a fresh coinstate |
| Resume | New node | Begin applying blocks from `snapshot_height + 1` |

### Verification

The Merkle root embedded in the snapshot MUST match the state root in the trusted block header at the snapshot height. If they do not match, the snapshot MUST be rejected.

### Checkpoint Selection

Snapshots SHOULD be created at regular intervals (e.g., every 10,000 blocks) or at operator-defined checkpoints.

---

## Acceptance Criteria

- [ ] Snapshot can be created from a live coinstate.
- [ ] Snapshot includes all coin records, Merkle state, chain tip, and counters.
- [ ] Snapshot includes a Merkle root.
- [ ] New node can restore a coinstate from a snapshot.
- [ ] Merkle root of restored state matches the snapshot's declared root.
- [ ] Restored node can resume block application from `snapshot_height + 1`.
- [ ] Snapshot with mismatched Merkle root is rejected.

---

## Implementation Notes

- **Primary file:** `src/snapshot.rs`
- **Dependencies:** Merkle tree (for root computation), storage layer (for serialization/deserialization), PRF-008 (persistence API)
- **Edge cases:**
  - Corrupted snapshot: Merkle root verification catches tampering.
  - Snapshot at height 0 (genesis): valid but trivial.
  - Snapshot taken during archival (PRF-005): archived coins may or may not be included depending on configuration.
  - Network distribution mechanism is out of scope for the coinstate crate.
