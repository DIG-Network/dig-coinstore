# PRF-001: In-Memory Unspent Set

> **Authoritative requirement:** [PRF-001](../NORMATIVE.md#PRF-001)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 13.2

## Summary

The coinstate maintains a `HashSet<CoinId>` in memory that mirrors the set of all currently-unspent coins. This enables O(1) `is_unspent()` lookups without hitting the storage layer, which is critical for removal validation during `apply_block` (BLK-005) and for parallel removal validation (CON-004).

Each `CoinId` is a 32-byte hash. With `HashSet` overhead (hash bucket pointer, capacity management), the per-entry cost is approximately 40 bytes. For a mainnet-scale coinstate with ~20 million unspent coins, this translates to roughly 800 MB of resident memory -- an acceptable trade-off for the performance gain.

On startup, the set is populated by scanning all unspent coins from the storage layer in batches of `MATERIALIZATION_BATCH_SIZE` (50,000) to avoid overwhelming memory with a single allocation. This batch-loading pattern also allows progress reporting during startup.

## Specification

### Data Structure

```rust
use std::collections::HashSet;

struct CoinStoreInner {
    unspent_set: HashSet<CoinId>,
    // ...
}
```

### Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `MATERIALIZATION_BATCH_SIZE` | 50,000 | Number of coins loaded per batch during startup population |

### Memory Budget

| Field | Size | Notes |
|-------|------|-------|
| `CoinId` (key) | 32 bytes | SHA-256 hash |
| HashSet overhead | ~8 bytes | Bucket pointer + metadata |
| **Total per entry** | **~40 bytes** | |

### Operations

| Operation | Trigger | Action |
|-----------|---------|--------|
| Insert | Coin creation (apply_block additions) | `unspent_set.insert(coin_id)` |
| Remove | Coin spend (apply_block removals) | `unspent_set.remove(&coin_id)` |
| Re-insert | Rollback of a spend | `unspent_set.insert(coin_id)` |
| Remove (rollback) | Rollback of a creation | `unspent_set.remove(&coin_id)` |
| Lookup | `is_unspent()` query | `unspent_set.contains(&coin_id)` |

### Startup Population

On startup, the implementation MUST:
1. Query storage for all unspent coin IDs.
2. Load them in batches of `MATERIALIZATION_BATCH_SIZE`.
3. Insert each batch into the `HashSet`.
4. Log progress (e.g., "Loaded 150,000 / 20,000,000 unspent coins").

## Acceptance Criteria

- [ ] `is_unspent()` performs an O(1) lookup against the in-memory `HashSet`.
- [ ] Coin creation inserts into the unspent set.
- [ ] Coin spend removes from the unspent set.
- [ ] Rollback of a spend re-inserts into the unspent set.
- [ ] Rollback of a creation removes from the unspent set.
- [ ] Startup populates the set from storage in `MATERIALIZATION_BATCH_SIZE` chunks.
- [ ] Per-entry memory consumption is approximately 40 bytes.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` or `src/unspent_set.rs`
- **Dependencies:** Storage layer (for startup population), BLK-005 (removal validation uses `is_unspent()`), CON-004 (parallel validation reads the set)
- **Edge cases:**
  - Empty coinstate on first startup: set is empty, no batches loaded.
  - Rollback to genesis: set should match the state after applying only block 0.
  - Concurrent reads during `apply_block`: the unspent set is mutated only during Phase 2 under an exclusive lock (CON-002).
