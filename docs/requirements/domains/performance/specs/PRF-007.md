# PRF-007: Height-Partitioned Indices

> **Authoritative requirement:** [PRF-007](../NORMATIVE.md#PRF-007)

## Summary

The `coin_by_confirmed_height` and `coin_by_spent_height` indices use a compound key that includes a height bucket prefix. This partitioning strategy exploits the LSM tree's natural compaction behavior: old height buckets are never written to again, so they settle into cold (deeper) LSM levels and do not interfere with writes or reads on recent heights.

This design provides efficient range scans on recent heights (the common case for monitoring and reorg detection) while keeping historical data accessible without requiring a separate archival mechanism for index entries.

## Specification

### Key Format

```
height_bucket || height || coin_id
```

| Component | Size | Description |
|-----------|------|-------------|
| `height_bucket` | 4 bytes | `height / BUCKET_SIZE` as big-endian u32 |
| `height` | 8 bytes | Block height as big-endian u64 |
| `coin_id` | 32 bytes | Coin ID (SHA-256 hash) |

### Column Families

| CF | Key Format | Value |
|----|------------|-------|
| `coin_by_confirmed_height` | `bucket \|\| confirmed_height \|\| coin_id` | `()` (empty) |
| `coin_by_spent_height` | `bucket \|\| spent_height \|\| coin_id` | `()` (empty) |

### Bucket Size

The bucket size determines the granularity of partitioning. A reasonable default is 10,000 blocks per bucket. This is a tuning parameter, not a consensus-critical value.

### LSM Compaction Behavior

- New blocks write to the current (highest) bucket.
- Once a bucket is full (no new writes), its SST files settle into deeper LSM levels during compaction.
- Range scans on recent heights only touch the top-level SST files.
- Historical range scans may touch deeper levels but are infrequent.

### Range Scan

To find all coins confirmed in a height range `[lo, hi]`:
1. Compute `bucket_lo = lo / BUCKET_SIZE` and `bucket_hi = hi / BUCKET_SIZE`.
2. For each bucket in `[bucket_lo, bucket_hi]`, scan the key range `[bucket||lo||0x00..., bucket||hi||0xFF...]`.

## Acceptance Criteria

- [ ] `coin_by_confirmed_height` uses compound key `bucket||height||coin_id`.
- [ ] `coin_by_spent_height` uses compound key `bucket||height||coin_id`.
- [ ] Keys are big-endian encoded for correct lexicographic ordering.
- [ ] Range scan on recent heights is efficient (touches few SST files).
- [ ] Old buckets settle into cold LSM levels after compaction.
- [ ] Range scan across bucket boundaries works correctly.

---

## Implementation Notes

- **Primary file:** `src/storage/rocksdb.rs` or `src/indices.rs`
- **Dependencies:** Storage layer (column family management, key encoding)
- **Edge cases:**
  - Height 0 is in bucket 0.
  - Range scan spanning multiple buckets must iterate across bucket boundaries.
  - Bucket size change requires re-indexing (should be a migration, not a runtime change).
  - LMDB backend may not benefit from this optimization (no LSM tree), but the key format can still be used for consistent range scan semantics.
