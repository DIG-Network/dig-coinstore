# PRF-003: Materialized Aggregate Counters

> **Authoritative requirement:** [PRF-003](../NORMATIVE.md#PRF-003)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 13.7

---

## Summary

Rather than scanning the entire coinstate to compute aggregate statistics (unspent count, spent count, total value), the implementation maintains materialized counters that are updated incrementally with each block. These counters live in the metadata column family and are updated in the same `WriteBatch` as the block's coin mutations, ensuring transactional consistency.

This enables O(1) reads for dashboard/monitoring queries that would otherwise require a full table scan over millions of records.

---

## Specification

### Counters

| Counter | Type | Description |
|---------|------|-------------|
| `unspent_count` | `u64` | Number of currently-unspent coins |
| `spent_count` | `u64` | Number of spent coins |
| `total_value` | `u64` | Sum of all unspent coin amounts (in mojos) |

### Storage

Counters MUST be stored in the metadata column family under well-known keys:

```
metadata:aggregate:unspent_count -> u64 (big-endian)
metadata:aggregate:spent_count   -> u64 (big-endian)
metadata:aggregate:total_value   -> u64 (big-endian)
```

### Update Rules

| Event | unspent_count | spent_count | total_value |
|-------|---------------|-------------|-------------|
| Coin created (addition) | +1 | unchanged | +coin.amount |
| Coin spent (removal) | -1 | +1 | -coin.amount |
| Rollback creation | -1 | unchanged | -coin.amount |
| Rollback spend | +1 | -1 | +coin.amount |

### Atomicity

Counter updates MUST be included in the same `WriteBatch` as the coin record mutations. If the batch fails, counters remain unchanged. If it succeeds, counters are consistent with the new coin state.

---

## Acceptance Criteria

- [ ] `unspent_count`, `spent_count`, and `total_value` are maintained in the metadata CF.
- [ ] Counters are updated in the same `WriteBatch` as block application.
- [ ] Counter reads are O(1) (single key lookup).
- [ ] Counters are consistent after `apply_block`.
- [ ] Counters are consistent after rollback.
- [ ] Counters survive restart (persisted in metadata CF).
- [ ] Counter values match a full scan of the coinstate.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` or `src/counters.rs`
- **Dependencies:** Storage layer (metadata CF), BLK-007/BLK-008 (counter updates happen alongside coin mutations)
- **Edge cases:**
  - Genesis block: counters start at 0, then increment for coinbase additions.
  - `total_value` uses `u64` to match the SPEC code example. Chia max supply is ~21M XCH = ~21 * 10^18 mojos, which fits comfortably in `u64`.
  - Counter drift: if counters ever drift from actual state (e.g., due to a bug), a repair tool should be able to recompute from a full scan.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| `test_counters_after_apply` | Integration | Apply block with 5 additions, 3 removals | unspent_count += 2, spent_count += 3, total_value adjusted |
| `test_counters_after_rollback` | Integration | Apply then rollback | Counters return to pre-block values |
| `test_counters_o1_read` | Unit | Read counters | Single key lookup, no scan |
| `test_counters_atomic_with_block` | Integration | Inject failure mid-WriteBatch | Counters unchanged on failure |
| `test_counters_survive_restart` | Integration | Apply block, restart, read counters | Counters persist |
| `test_counters_match_scan` | Integration | Apply several blocks, compare counters to full scan | Values match |
| `test_counters_after_partial_rollback` | Integration | Apply blocks 1-5, rollback to block 3 | Counters reflect state at block 3 exactly |
| `test_counters_after_fork` | Integration | Apply blocks 1-5, rollback to 3, apply alternative blocks 4-5 | Counters reflect the new fork chain state |
| `test_counters_with_coinbase` | Integration | Apply block with coinbase coins | Counters correctly increment for coinbase additions |
