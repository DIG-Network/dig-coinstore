## Summary

`CoinStoreSnapshot` is a serializable representation of the complete coinstate at a point in time. It is used for snapshot/restore operations, fast sync for new nodes, and backup/recovery.

The snapshot contains all coin records, all hints, chain metadata (height, block hash, state root, timestamp), and aggregate statistics (total coins, total value). It derives `Serialize` and `Deserialize` for persistence via bincode or other serde-compatible formats.

This is a dig-coinstore extension not present in Chia's `CoinStore` (SPEC Section 1.6 Improvement #6). Chia has no built-in snapshot mechanism; new nodes must replay the entire chain from genesis.

## Specification

### Struct Definition

```rust
#[derive(Serialize, Deserialize)]
pub struct CoinStoreSnapshot {
    /// Chain tip height at snapshot time.
    pub height: u64,
    /// Chain tip block hash.
    pub block_hash: Bytes32,
    /// State root at snapshot time.
    pub state_root: Bytes32,
    /// Timestamp at snapshot time.
    pub timestamp: u64,
    /// All coin records.
    pub coins: HashMap<CoinId, CoinRecord>,
    /// All hints.
    pub hints: Vec<(CoinId, Bytes32)>,
    /// Total coins.
    pub total_coins: u64,
    /// Total unspent value.
    pub total_value: u64,
}
```

### Derive Requirements

- `Serialize`: Required for snapshot persistence (writing to disk or network).
- `Deserialize`: Required for snapshot restore (reading from disk or network).
- `CoinRecord` must also derive `Serialize, Deserialize` (see [API-002](API-002.md)).
- `Bytes32` from `chia-protocol` must support serde (it does).
- `HashMap` and `Vec` are serde-compatible by default.

### Related Methods

```rust
impl CoinStore {
    /// Take a full snapshot of the current coinstate.
    pub fn snapshot(&self) -> Result<CoinStoreSnapshot, CoinStoreError>;

    /// Restore coinstate from a snapshot. Replaces all current state.
    pub fn restore(&mut self, snapshot: CoinStoreSnapshot) -> Result<(), CoinStoreError>;

    /// Save a snapshot to persistent storage (keyed by height).
    /// Automatically prunes old snapshots beyond `max_snapshots`.
    pub fn save_snapshot(&self) -> Result<(), CoinStoreError>;

    /// Load a snapshot from persistent storage by height.
    pub fn load_snapshot(&self, height: u64) -> Result<Option<CoinStoreSnapshot>, CoinStoreError>;

    /// Load the most recent snapshot from persistent storage.
    pub fn load_latest_snapshot(&self) -> Result<Option<CoinStoreSnapshot>, CoinStoreError>;

    /// Get available snapshot heights.
    pub fn available_snapshot_heights(&self) -> Vec<u64>;
}
```

### Field Sources

| Field | Source | Description |
|-------|--------|-------------|
| `height` | Chain metadata | Tip height when snapshot was taken |
| `block_hash` | Chain metadata | Tip block hash when snapshot was taken |
| `state_root` | Merkle tree | Root hash when snapshot was taken |
| `timestamp` | Chain metadata | Tip timestamp when snapshot was taken |
| `coins` | Coin store scan | All coin records (spent + unspent) |
| `hints` | Hint store scan | All (coin_id, hint) pairs |
| `total_coins` | `coins.len()` | Total records in snapshot |
| `total_value` | Aggregation | Sum of unspent coin amounts |

### Snapshot Retention

Snapshots are automatically pruned when the count exceeds `CoinStoreConfig::max_snapshots` (default 10). Oldest snapshots (by height) are deleted first. See [API-003](API-003.md).

### Restore Semantics

`restore()` replaces all current coinstate with the snapshot's data:
1. Clear all existing coin records, indices, and hints.
2. Insert all coins from `snapshot.coins`.
3. Rebuild all secondary indices (puzzle hash, height, parent).
4. Insert all hints from `snapshot.hints`.
5. Rebuild the Merkle tree and verify the root matches `snapshot.state_root`.
6. Set chain metadata (height, block_hash, timestamp).
7. If the Merkle root does not match, return `Err(CoinStoreError::StateRootMismatch)`.

## Implementation Notes

- **Primary files:** `src/types.rs` (`CoinStoreSnapshot`), `src/coin_store.rs` (`snapshot` / `restore` / retention APIs)
- **Serialization format:** Use `bincode` for compact binary representation. Snapshots can be large (millions of coins), so compact encoding is important.
- **Batched materialization:** When restoring, coins should be inserted in batches of `MATERIALIZATION_BATCH_SIZE` (50,000) to prevent OOM on large coin sets (SPEC Section 2.7).
- **State root verification:** After restoring, the Merkle tree is rebuilt from scratch and the computed root is compared to `snapshot.state_root`. This ensures the snapshot is internally consistent.
- **Edge cases:**
  - Empty snapshot (height 0, no coins): valid, represents pre-genesis state.
  - Snapshot with spent coins: valid, spent coins are preserved for historical queries.
  - Restoring over existing data: all existing data is cleared first.
