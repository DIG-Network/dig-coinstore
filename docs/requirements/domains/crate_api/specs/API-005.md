## Summary

`BlockData` is the input to `CoinStore::apply_block()`. It represents the pre-validated, pre-extracted set of state changes that the caller has already derived from CLVM execution. The coinstate does not run CLVM; it trusts the caller to provide correct additions and removals.

`CoinAddition` is a companion struct that wraps a coin with its metadata, including the `same_as_parent` flag used for singleton fast-forward eligibility tracking.

These types correspond to the parameters of Chia's `CoinStore.new_block()` ([`coin_store.py:105-178`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/coin_store.py#L105)): `height`, `timestamp`, `included_reward_coins`, `tx_additions` (list of tuples `(coin, CoinId, same_as_parent)`), `tx_removals` (list of CoinId bytes).

## Specification

### BlockData Struct Definition

```rust
/// Pre-validated block state changes for coinstate application.
pub struct BlockData {
    /// Block height (must be exactly `current_height + 1`).
    pub height: u64,
    /// Block timestamp (unix seconds).
    pub timestamp: u64,
    /// Hash of this block's header (for chain tracking).
    pub block_hash: Bytes32,
    /// Hash of the parent block's header (must match current chain tip).
    pub parent_hash: Bytes32,
    /// Coins created by transactions in this block.
    /// Each entry: (coin_id, coin, is_same_puzzle_as_parent).
    /// Chia: `tx_additions` parameter in coin_store.py:121.
    pub additions: Vec<CoinAddition>,
    /// Coin IDs destroyed (spent) by transactions in this block.
    /// Chia: `tx_removals` parameter in coin_store.py:122.
    pub removals: Vec<CoinId>,
    /// Coinbase reward coins created at this height.
    /// Chia: `included_reward_coins` parameter in coin_store.py:110.
    pub coinbase_coins: Vec<Coin>,
    /// Hints extracted from CREATE_COIN conditions (coin_id -> hint bytes).
    /// Stored in the hint index for wallet-style lookups.
    /// Chia: processed by hint_management.py and stored in HintStore.
    pub hints: Vec<(CoinId, Bytes32)>,
    /// Expected state root after applying this block.
    /// If provided, the coinstate verifies its computed root matches. If it does not
    /// match, the block is rejected with StateRootMismatch.
    pub expected_state_root: Option<Bytes32>,
}
```

### CoinAddition Struct Definition

```rust
/// A coin addition with metadata.
pub struct CoinAddition {
    /// The computed coin ID.
    pub coin_id: CoinId,
    /// The coin being created.
    pub coin: Coin,
    /// Whether this coin has the same puzzle hash and amount as its parent.
    /// Used for singleton fast-forward unspent tracking.
    /// Chia: `same_as_parent` flag in tx_additions (coin_store.py:121).
    pub same_as_parent: bool,
}
```

### Field Sources

| Field | Source | Description |
|-------|--------|-------------|
| `height` | Block header | Must be sequential: `current_height + 1` |
| `timestamp` | Block header | Unix seconds of block creation |
| `block_hash` | Block header hash | SHA256 of serialized header |
| `parent_hash` | Block header | Must match current `tip_hash()` |
| `additions` | CLVM execution | Coins from CREATE_COIN conditions |
| `removals` | CLVM execution | Coin IDs from spent coin spends |
| `coinbase_coins` | Block rewards | Farmer + pool reward coins |
| `hints` | CLVM execution | Hints from CREATE_COIN conditions |
| `expected_state_root` | Block header / caller | Optional verification of computed root |
| `coin_id` | Derived | `sha256(parent_coin_info \|\| puzzle_hash \|\| amount)` |
| `coin` | CLVM execution | The Coin being created |
| `same_as_parent` | Caller analysis | Parent shares puzzle_hash and amount |

### Validation Rules

`BlockData` is validated by `apply_block()` before any state mutation:

1. `height == current_height + 1` (HeightMismatch)
2. `parent_hash == current tip_hash` (ParentHashMismatch)
3. For height 0: `coinbase_coins` must be empty. For height > 0: `coinbase_coins.len() >= 2` (InvalidRewardCoinCount)
4. All removals must reference existing, unspent coins (CoinNotFound, DoubleSpend)
5. No addition coin_id may already exist (CoinAlreadyExists)
6. All hints must be <= MAX_HINT_LENGTH bytes (HintTooLong)
7. If `expected_state_root` is Some, computed root must match (StateRootMismatch)

## Implementation Notes

- **Primary file:** `src/types.rs` or `src/block.rs`
- **Type imports:** `Coin`, `Bytes32` from `chia-protocol`. `CoinId` is `Bytes32` alias.
- **Hard boundary:** `BlockData` does NOT contain CLVM programs, signatures, or raw block bytes. The caller extracts additions/removals/hints from CLVM execution results before constructing `BlockData`.
- **same_as_parent computation:** The caller compares the addition's `puzzle_hash` and `amount` to its parent coin's `puzzle_hash` and `amount`. This requires the caller to have access to the parent coin record.
- **Hint storage:** Hints are stored in a dedicated index. Empty hints (0-length) in the source data are silently skipped during storage. Only 32-byte hints are eligible for puzzle-hash subscription matching.
