## Summary

The `Cargo.toml` file defines the crate identity, dependencies, and feature gates for `dig-coinstore`. This crate is extracted from the working coin state management code in `l2_driver_state_channel` and serves as the standalone, reusable coin state database for the DIG L2 network.

The crate name MUST be `dig-coinstore` with a `lib` crate type. All external dependencies required for coin state management -- Chia protocol types, storage backends, serialization, concurrency, and diagnostics -- MUST be declared. Storage backends (RocksDB and LMDB) MUST be gated behind Cargo features so consumers can select their preferred backend at compile time without pulling in unnecessary native dependencies.

## Specification

### Crate Identity

```toml
[package]
name = "dig-coinstore"
version = "0.1.0"
edition = "2021"

[lib]
name = "dig_coinstate"
```

### Required Dependencies

| Dependency | Purpose |
|-----------|---------|
| `chia-protocol` | Chia blockchain protocol types (`Coin`, `CoinState`, `CoinStateFilters`, `Bytes32`, `CoinRecord` for interop) |
| `chia-sha2` | SHA-256 implementation (`Sha256` hasher) for Merkle leaf/node hashing. Same impl used by `Coin::coin_id()`. |
| `chia-traits` | `Streamable` trait for Chia-canonical wire-format serialization of protocol types like `CoinState` |
| `dig-clvm` | DIG CLVM integration, re-exports Chia types |
| `dig-constants` | DIG network constants (puzzle hashes, launcher IDs) |
| `rocksdb` | RocksDB storage backend (feature-gated) |
| `heed` | LMDB storage backend via heed bindings (feature-gated) |
| `bincode` | Binary serialization for internal storage values |
| `serde` | Serialization/deserialization derive support |
| `parking_lot` | Fast RwLock/Mutex for interior mutability |
| `thiserror` | Ergonomic error enum derivation |
| `tracing` | Structured logging and diagnostics |

### Dev Dependencies

| Dependency | Purpose |
|-----------|---------|
| `chia-consensus` | `compute_merkle_set_root()`, `MerkleSet` for cross-checking Merkle computations in tests |
| `chia-sdk-test` | `Simulator` in-memory coin store used as oracle for query parity integration tests |

### Feature Gates

```toml
[features]
default = ["rocksdb-storage"]
rocksdb-storage = ["rocksdb"]
lmdb-storage = ["heed"]
full-storage = ["rocksdb-storage", "lmdb-storage"]
```

- `rocksdb-storage` -- Enables the RocksDB backend. This is the default.
- `lmdb-storage` -- Enables the LMDB backend.
- `full-storage` -- Enables both backends simultaneously (useful for testing or migration).

## Acceptance Criteria

- [ ] `Cargo.toml` declares `name = "dig-coinstore"` under `[package]`.
- [ ] `[lib]` section specifies `name = "dig_coinstate"`.
- [ ] All twelve required dependencies are declared in `[dependencies]` or feature-conditional sections (`chia-protocol`, `chia-sha2`, `chia-traits`, `dig-clvm`, `dig-constants`, `rocksdb`, `heed`, `bincode`, `serde`, `parking_lot`, `thiserror`, `tracing`).
- [ ] `chia-consensus` and `chia-sdk-test` are declared in `[dev-dependencies]`.
- [ ] `rocksdb` dependency is gated behind `rocksdb-storage` feature.
- [ ] `heed` dependency is gated behind `lmdb-storage` feature.
- [ ] `full-storage` feature activates both `rocksdb-storage` and `lmdb-storage`.
- [ ] Default feature is `rocksdb-storage`.
- [ ] `cargo check --features rocksdb-storage` succeeds.
- [ ] `cargo check --features lmdb-storage` succeeds.
- [ ] `cargo check --features full-storage` succeeds.
- [ ] `cargo check --no-default-features` succeeds (compiles without any storage backend).
- [ ] `chia-sha2::Sha256` is importable and usable from within the crate.

---

## Implementation Notes

- **Primary file:** `Cargo.toml` (crate root)
- **Dependencies:** The `rocksdb` and `heed` crates pull in native C/C++ libraries. Feature gating ensures consumers only compile the backends they need.
- **Edge cases:**
  - Building with `--no-default-features` should still compile the core types and trait definitions, even though no storage backend is available.
  - The `dig-clvm` and `dig-constants` crates are workspace-local path dependencies in the DIG monorepo.
