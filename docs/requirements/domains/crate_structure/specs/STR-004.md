## Summary

The merkle module implements a sparse Merkle tree (SMT) for maintaining a cryptographic commitment to the coin set state. The `SparseMerkleTree` struct in `src/merkle/mod.rs` provides batch operations for inserting, updating, and removing leaves, and exposes the current root hash. Proof generation and verification are handled in `src/merkle/proof.rs`, enabling third parties to verify coin inclusion or exclusion without the full dataset. The persistent layer in `src/merkle/persistent.rs` handles loading and flushing dirty tree nodes to the storage backend.

This module is critical for the DIG L2 network's ability to produce verifiable state commitments. Each block application updates the SMT, and the resulting root hash is included in the block header.

## Specification

### SparseMerkleTree Struct

`src/merkle/mod.rs` MUST define a `SparseMerkleTree` struct with the following public methods:

| Method | Signature | Description |
|--------|-----------|-------------|
| `batch_insert` | `fn batch_insert(&mut self, entries: &[(Key, Value)]) -> Result<(), MerkleError>` | Insert multiple new leaves into the tree |
| `batch_update` | `fn batch_update(&mut self, entries: &[(Key, Value)]) -> Result<(), MerkleError>` | Update existing leaves with new values |
| `batch_remove` | `fn batch_remove(&mut self, keys: &[Key]) -> Result<(), MerkleError>` | Remove leaves from the tree |
| `root` | `fn root(&self) -> Bytes32` | Return the current Merkle root hash |

### Proof Module

`src/merkle/proof.rs` MUST define:
- `SparseMerkleProof` struct containing the sibling hashes and path information needed to verify membership.
- `verify()` method on `SparseMerkleProof` that takes a root hash, key, and optional value, and returns whether the proof is valid.
- Support for both inclusion proofs (key exists with value) and exclusion proofs (key does not exist).

### Persistent Node Storage

`src/merkle/persistent.rs` MUST implement:
- Loading tree nodes from the storage backend on demand (lazy loading).
- Tracking dirty (modified) nodes during batch operations.
- Flushing all dirty nodes to storage in a single batch write.
- Integration with the `StorageBackend` trait from the storage module.

## Acceptance Criteria

- [ ] `src/merkle/mod.rs` defines a public `SparseMerkleTree` struct.
- [ ] `batch_insert`, `batch_update`, `batch_remove`, and `root()` methods exist with correct signatures.
- [ ] `src/merkle/proof.rs` defines a public `SparseMerkleProof` struct.
- [ ] `SparseMerkleProof` has a `verify()` method.
- [ ] Inclusion and exclusion proofs are supported.
- [ ] `src/merkle/persistent.rs` implements lazy node loading from storage.
- [ ] Dirty nodes are tracked and can be flushed in a batch.
- [ ] The persistent layer uses `StorageBackend` (not a concrete backend directly).
- [ ] An empty tree has a well-defined root hash (all zeros or a defined constant).
- [ ] Batch operations are atomic -- all succeed or the tree state is unchanged.

## Implementation Notes

- **Primary files:** `src/merkle/mod.rs`, `src/merkle/proof.rs`, `src/merkle/persistent.rs`
- **Dependencies:** STR-003 (storage backend for persistent nodes), STR-002 (file layout)
- **Edge cases:**
  - Inserting a key that already exists should be handled (error or update semantics).
  - Removing a key that does not exist should be handled gracefully.
  - Very deep trees (256-bit keys) require efficient path compression to avoid excessive memory usage.
  - The root of an empty tree must be deterministic and consistent across restarts.
