## Summary

The storage module provides an abstraction layer over the underlying key-value database engines used by `dig-coinstore`. The `StorageBackend` trait, defined in `src/storage/mod.rs`, establishes a uniform interface for both RocksDB and LMDB (via heed) backends. This allows the rest of the crate to be storage-agnostic, with the concrete backend selected at compile time through Cargo features.

Each backend implementation lives in its own file (`rocksdb.rs` and `lmdb.rs`) and MUST implement the full `StorageBackend` trait. A shared schema module (`schema.rs`) defines column family names and key encoding/decoding helpers that are consistent across both backends.

---

## Specification

### StorageBackend Trait

`src/storage/mod.rs` MUST define a `StorageBackend` trait with methods for:
- Opening/initializing the database at a given path.
- Reading and writing key-value pairs within named column families.
- Batch write operations (atomic multi-key writes).
- Column family management (creation, listing).
- Transaction/snapshot support for consistent reads.

### RocksDB Implementation

`src/storage/rocksdb.rs` MUST:
1. Be conditionally compiled with `#[cfg(feature = "rocksdb-storage")]`.
2. Implement `StorageBackend` for a `RocksDbBackend` struct.
3. Use the `rocksdb` crate for all database operations.
4. Support column families as defined in `schema.rs`.

### LMDB Implementation

`src/storage/lmdb.rs` MUST:
1. Be conditionally compiled with `#[cfg(feature = "lmdb-storage")]`.
2. Implement `StorageBackend` for an `LmdbBackend` struct.
3. Use the `heed` crate for all database operations.
4. Map column family concepts to LMDB named databases.

### Schema Module

`src/storage/schema.rs` MUST:
1. Define string constants for all column family names used by the coin store.
2. Provide key encoding helpers (e.g., `coin_id_to_key`, `height_to_key`).
3. Provide key decoding helpers (reverse of encoding).
4. Be usable by both backend implementations.

## Acceptance Criteria

- [ ] `src/storage/mod.rs` defines a public `StorageBackend` trait.
- [ ] The trait includes methods for open, get, put, delete, batch write, and iteration.
- [ ] `src/storage/rocksdb.rs` exists and is gated behind `#[cfg(feature = "rocksdb-storage")]`.
- [ ] `RocksDbBackend` implements `StorageBackend`.
- [ ] `src/storage/lmdb.rs` exists and is gated behind `#[cfg(feature = "lmdb-storage")]`.
- [ ] `LmdbBackend` implements `StorageBackend`.
- [ ] `src/storage/schema.rs` defines column family name constants.
- [ ] `src/storage/schema.rs` provides key encoding/decoding helper functions.
- [ ] Both backends compile independently with their respective features.
- [ ] Both backends pass the same trait-level test suite.

## Implementation Notes

- **Primary files:** `src/storage/mod.rs`, `src/storage/rocksdb.rs`, `src/storage/lmdb.rs`, `src/storage/schema.rs`
- **Dependencies:** STR-001 (feature gates and dependencies), STR-002 (file layout)
- **Edge cases:**
  - When neither storage feature is enabled, `mod.rs` should still define the trait (for type checking) but no concrete implementation will be available.
  - LMDB has a fixed maximum database size that must be configured at open time. The `LmdbBackend` should accept this as a configuration parameter.
  - RocksDB column families must be created before first use. The `RocksDbBackend` open method should handle initial column family creation.
