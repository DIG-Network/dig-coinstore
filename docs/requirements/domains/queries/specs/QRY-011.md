# QRY-011: Input Slice Chunking for Large Batches

> **Authoritative requirement:** [QRY-011](../NORMATIVE.md#QRY-011)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

All query methods that accept `&[T]` input slices MUST chunk large inputs into batches of `DEFAULT_LOOKUP_BATCH_SIZE` to prevent unbounded memory allocation. Without chunking, a caller passing millions of coin IDs or puzzle hashes in a single call could cause the storage backend to allocate an equally large number of intermediate results, leading to OOM conditions. This is adopted from Chia's `to_batches()` pattern used in `coin_store.py`.

---

## Specification

### Constant

```rust
/// Maximum number of items per internal lookup batch.
/// Adopted from Chia's DEFAULT_LOOKUP_BATCH_SIZE.
pub const DEFAULT_LOOKUP_BATCH_SIZE: usize = 1000;
```

### Affected Methods

The following query methods MUST chunk their input slices:

| Method | Input Parameter |
|--------|----------------|
| `get_coin_records(coin_ids)` | `coin_ids: &[Bytes32]` |
| `get_coin_records_by_names(names)` | `names: &[Bytes32]` |
| `get_coin_records_by_puzzle_hashes(puzzle_hashes)` | `puzzle_hashes: &[Bytes32]` |
| `get_coin_records_by_parent_ids(parent_ids)` | `parent_ids: &[Bytes32]` |
| `get_coin_ids_by_hints(hints)` | `hints: &[Bytes32]` |

### Chunking Algorithm

```rust
fn chunked_lookup<T, R>(
    items: &[T],
    batch_size: usize,
    lookup_fn: impl Fn(&[T]) -> Result<Vec<R>, CoinStoreError>,
) -> Result<Vec<R>, CoinStoreError> {
    let mut results = Vec::new();
    for chunk in items.chunks(batch_size) {
        results.extend(lookup_fn(chunk)?);
    }
    Ok(results)
}
```

### Behavior

1. If the input slice length is less than or equal to `DEFAULT_LOOKUP_BATCH_SIZE`, the method proceeds without chunking (single batch).
2. If the input slice length exceeds `DEFAULT_LOOKUP_BATCH_SIZE`, the method splits the input into chunks of `DEFAULT_LOOKUP_BATCH_SIZE` elements and processes each chunk independently, accumulating results.
3. Error handling: if any chunk fails, the entire call fails immediately (no partial results).
4. Deduplication (where specified by the underlying method, e.g., QRY-001 batch) MUST still be applied across all chunks, not just within each chunk.
5. The chunking is an internal implementation detail; the caller-facing API does not change.

---

## Acceptance Criteria

- [ ] `DEFAULT_LOOKUP_BATCH_SIZE` constant is defined with value 1000.
- [ ] All five listed methods chunk inputs when slice length exceeds `DEFAULT_LOOKUP_BATCH_SIZE`.
- [ ] Methods with input slices at or below the batch size execute without chunking overhead.
- [ ] Results are identical regardless of whether chunking was triggered.
- [ ] Deduplication is applied across all chunks where the method specifies dedup behavior.
- [ ] An error in any chunk propagates immediately; no partial results are returned.

## Implementation Notes

- **Primary file:** `src/queries.rs`
- **Dependencies:** QRY-001, QRY-002, QRY-004, QRY-005, HNT-004
- **Edge cases:**
  - Empty input slice: returns empty result, no chunks processed.
  - Input length exactly equal to `DEFAULT_LOOKUP_BATCH_SIZE`: single chunk, no split.
  - Input length = `DEFAULT_LOOKUP_BATCH_SIZE + 1`: two chunks (one full, one with 1 item).
  - Cross-chunk dedup: if the same coin ID appears in two chunks, dedup must merge them.
