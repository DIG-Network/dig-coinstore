# QRY-001: Point Lookups by Coin ID

> **Authoritative requirement:** [QRY-001](../NORMATIVE.md#QRY-001)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

Point lookups are the most fundamental query operation in the coin store. `get_coin_record()` retrieves a single coin record by its coin ID, while `get_coin_records()` performs a batch lookup for multiple coin IDs in one call. These are used throughout the system for coin validation, spend verification, and state queries.

This mirrors Chia L1's `get_coin_record()` and `get_coin_records()` methods in `coin_store.py`.

---

## Specification

### Function Signatures

```rust
impl CoinStore {
    pub fn get_coin_record(&self, coin_id: &Bytes32) -> Result<Option<CoinRecord>, CoinStoreError>;
    pub fn get_coin_records(&self, coin_ids: &[Bytes32]) -> Result<Vec<CoinRecord>, CoinStoreError>;
}
```

### Behavior

- `get_coin_record()`: Returns `Some(CoinRecord)` if the coin exists, `None` if not found.
- `get_coin_records()`: Returns records for all found coin IDs. Missing coin IDs are silently skipped (no error). The returned vector MAY be in any order.

### Performance

- Single lookups SHOULD use the LRU cache (PRF-002) before hitting storage.
- Batch lookups SHOULD use multi-get operations on the storage backend.

## Acceptance Criteria

- [ ] `get_coin_record()` returns `Some(record)` for existing coins.
- [ ] `get_coin_record()` returns `None` for non-existent coins.
- [ ] `get_coin_records()` returns records for all found IDs.
- [ ] `get_coin_records()` silently skips missing IDs.
- [ ] Both methods return spent and unspent coins.
- [ ] LRU cache is consulted before storage (PRF-002).

---

## Implementation Notes

- **Primary file:** `src/queries.rs`
- **Dependencies:** STO-001 (storage get), PRF-002 (LRU cache)
- **Edge cases:**
  - Empty `coin_ids` slice returns empty vector.
  - Duplicate IDs in batch: return one record per unique ID.
