# QRY-008: Singleton Lineage Lookup

> **Authoritative requirement:** [QRY-008](../NORMATIVE.md#QRY-008)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

`get_unspent_lineage_info_for_puzzle_hash()` retrieves singleton fast-forward lineage information for a given puzzle hash. This is used by the fast-forward (FF) optimization for singletons, where a coin's spend can be replayed without full CLVM re-execution if the singleton lineage is intact. The function returns `None` if the puzzle hash does not match exactly one unspent coin, since FF only works for singletons (exactly one live coin per puzzle hash).

This mirrors Chia L1's `get_unspent_lineage_info_for_puzzle_hash()` method.

---

## Specification

### Function Signature

```rust
impl CoinStore {
    pub fn get_unspent_lineage_info_for_puzzle_hash(
        &self,
        puzzle_hash: &Bytes32,
    ) -> Result<Option<UnspentLineageInfo>, CoinStoreError>;
}
```

### Behavior

1. Query the `unspent_by_puzzle_hash` index for the given puzzle hash.
2. If the number of matching unspent coins is not exactly 1, return `None`.
3. If exactly 1 unspent coin matches, look up the coin and its parent to construct and return `UnspentLineageInfo` containing the coin ID, parent ID, and parent's parent ID.

### UnspentLineageInfo Struct (SPEC Section 2.5)

```rust
pub struct UnspentLineageInfo {
    /// The coin ID of the singleton.
    pub coin_id: Bytes32,
    /// The parent coin ID of the singleton.
    pub parent_id: Bytes32,
    /// The grandparent coin ID (parent's parent). `None` if the parent has no
    /// recorded parent (e.g., the parent is not in the coin store).
    pub parent_parent_id: Option<Bytes32>,
}
```

---

## Acceptance Criteria

- [ ] Returns `Some(UnspentLineageInfo)` when exactly one unspent coin matches the puzzle hash.
- [ ] Returns `None` when zero unspent coins match.
- [ ] Returns `None` when two or more unspent coins match.
- [ ] `UnspentLineageInfo` contains correct `coin_id`, `parent_id`, and `parent_parent_id` fields per SPEC Section 2.5.

## Implementation Notes

- **Primary file:** `src/queries.rs`
- **Dependencies:** PRF-004 (unspent-only puzzle hash index)
- **Edge cases:**
  - Puzzle hash with many spent coins but exactly one unspent: returns Some.
  - Freshly created singleton (no parent in store): `parent_parent_id` is `None`.
