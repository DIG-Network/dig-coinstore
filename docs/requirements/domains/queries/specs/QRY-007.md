# QRY-007: Batch Coin State Pagination

> **Authoritative requirement:** [QRY-007](../NORMATIVE.md#QRY-007)

> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

`batch_coin_states_by_puzzle_hashes()` is the most complex query in the coin store, designed for the wallet sync protocol's subscription-based coin state streaming. It supports paginated retrieval of coin states across multiple puzzle hashes with hint-based inclusion, deduplication, minimum amount filtering, deterministic sorting, and block boundary preservation to ensure consistent pagination.

This mirrors Chia L1's `batch_coin_states_by_puzzle_hashes()` which is the core of the wallet sync v2 protocol.

## Specification

### Function Signature

```rust
/// SPEC Section 3.5
impl CoinStore {
    pub fn batch_coin_states_by_puzzle_hashes(
        &self,
        puzzle_hashes: &[Bytes32],
        min_height: u32,
        include_spent: bool,
        include_unspent: bool,
        include_hinted: bool,
        min_amount: Option<u64>,
        max_items: usize,
    ) -> Result<(Vec<CoinState>, Option<u64>), CoinStoreError>;
}
```

### Constraints and Behavior

1. **Batch size limit:** `puzzle_hashes.len()` MUST NOT exceed `MAX_PUZZLE_HASH_BATCH_SIZE`. If exceeded, return an error.
2. **Spent/unspent filtering:** If `include_spent` is `false`, spent coins MUST be excluded. If `include_unspent` is `false`, unspent coins MUST be excluded. At least one of the two MUST be `true`.
3. **Hinted coin inclusion:** If `include_hinted` is `true`, coins hinted to any of the provided puzzle hashes MUST be included in results (join with hints index), then deduplicated by coin ID. If `include_hinted` is `false`, only direct puzzle hash matches are returned.
4. **Min amount filter:** If `min_amount` is `Some(v)`, coins with `amount < v` MUST be excluded. `min_amount` of `Some(0)` is equivalent to no filter.
5. **Deterministic sort:** Results MUST be sorted by `MAX(confirmed_height, spent_height)` ASC. Ties MUST be broken deterministically (e.g., by coin ID).
6. **Block boundary preservation:** The result MUST NOT split a block across pages. If including all coins from the last block would exceed `max_items`, truncate before that block begins.
7. **Pagination detection:** Fetch `max_items + 1` internally. If `max_items + 1` items are available, the second element of the returned tuple is `Some(next_height)` where `next_height` is the height of the first item that did not fit. If all items fit, the second element is `None`.

### Return Type

The return type is `(Vec<CoinState>, Option<u64>)` matching SPEC Section 3.5:

| Element | Description |
|---------|-------------|
| `Vec<CoinState>` | The page of coin states, truncated at block boundaries. |
| `Option<u64>` | `Some(next_height)` if more pages remain, `None` if this is the last page. |

This replaces the previous `BatchCoinStateResult` struct to align with the SPEC.

## Acceptance Criteria

- [ ] Enforces `MAX_PUZZLE_HASH_BATCH_SIZE` limit.
- [ ] `include_spent=false` excludes spent coins; `include_unspent=false` excludes unspent coins.
- [ ] `include_hinted=true` includes hinted coins via hint index join; `include_hinted=false` returns direct matches only.
- [ ] Deduplicates by coin ID after merging direct and hinted matches.
- [ ] `min_amount` filter excludes coins below threshold; `Some(0)` is equivalent to no filter.
- [ ] Sort order is `MAX(confirmed_height, spent_height)` ASC with deterministic tiebreaking.
- [ ] Block boundaries are preserved (no partial blocks at page end).
- [ ] Returns `(Vec<CoinState>, Option<u64>)` per SPEC Section 3.5.
- [ ] Second tuple element is `Some(next_height)` when more pages remain, `None` when all items fit.

## Implementation Notes

- **Primary file:** `src/queries.rs`
- **Dependencies:** HNT-004 (hint queries), QRY-006 (CoinState), STO-002/STO-003 (indices)
- **Edge cases:**
  - All coins in a single block: if count > max_items, return empty (cannot split the block) with `Some(block_height)` as cursor.
  - No matching coins: return empty with `None`.
  - min_amount = Some(0) is equivalent to no filter.
  - Empty puzzle_hashes input: return empty with `None`.
  - `include_hinted=false`: only direct puzzle hash matches, no hint join.
