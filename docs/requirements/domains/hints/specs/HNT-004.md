# HNT-004: Hint Queries

## Summary

The hint store exposes a set of query functions that enable callers to look up coins by hint, look up hints by coin, perform batch queries across multiple hints, and obtain a total hint count. These queries are the primary public interface through which wallets and the RPC layer discover coins via puzzle-hash hints.

This mirrors the query methods in Chia L1's `HintStore` class, adapted from SQL to KV prefix-scan semantics.

## Specification

### Function Signatures

```rust
impl HintStore {
    /// Returns coin IDs associated with a single hint, up to max_items.
    pub fn get_coin_ids_by_hint(
        &self,
        hint: &[u8; 32],
        max_items: usize,
    ) -> Result<Vec<CoinId>, HintError>;

    /// Returns coin IDs associated with any of the provided hints, up to max_items total.
    pub fn get_coin_ids_by_hints(
        &self,
        hints: &[[u8; 32]],
        max_items: usize,
    ) -> Result<Vec<CoinId>, HintError>;

    /// Returns all hints associated with the given coin IDs.
    pub fn get_hints_for_coin_ids(
        &self,
        coin_ids: &[CoinId],
    ) -> Result<HashMap<CoinId, Vec<[u8; 32]>>, HintError>;

    /// Returns the total number of hint entries in the store.
    pub fn count_hints(&self) -> Result<u64, HintError>;
}
```

### Behavior

#### `get_coin_ids_by_hint(hint, max_items)`

1. Prefix-scan the reverse index with `hint` as the 32-byte prefix.
2. Extract coin IDs from the trailing 32 bytes of each matching key.
3. Stop after collecting `max_items` coin IDs.
4. Return the collected coin IDs in insertion order.

#### `get_coin_ids_by_hints(hints, max_items)`

1. For each hint in `hints`, perform the same reverse-index prefix scan.
2. Aggregate results across all hints into a single `Vec<CoinId>`.
3. Deduplicate coin IDs that appear under multiple hints.
4. Stop after collecting `max_items` total coin IDs.
5. The order of iteration across hints is unspecified; callers MUST NOT depend on ordering.

#### `get_hints_for_coin_ids(coin_ids)`

1. For each coin_id in `coin_ids`, prefix-scan the forward index with `coin_id` as the 32-byte prefix.
2. Extract hints from the trailing 32 bytes of each matching key.
3. Return a map from each coin_id to its list of hints.
4. Coin IDs with no hints SHOULD be omitted from the result map.

#### `count_hints()`

1. Return the total number of entries in the forward index.
2. This MAY use a materialized counter for O(1) performance instead of scanning.

## Acceptance Criteria

- [ ] `get_coin_ids_by_hint` returns correct coin IDs for a given hint.
- [ ] `get_coin_ids_by_hint` respects `max_items` limit.
- [ ] `get_coin_ids_by_hint` returns empty `Vec` for unknown hint.
- [ ] `get_coin_ids_by_hints` aggregates results across multiple hints.
- [ ] `get_coin_ids_by_hints` deduplicates coin IDs appearing under multiple hints.
- [ ] `get_coin_ids_by_hints` respects `max_items` total limit.
- [ ] `get_hints_for_coin_ids` returns correct hints for each coin.
- [ ] `get_hints_for_coin_ids` omits coin IDs with no hints.
- [ ] `count_hints` returns the total number of hint entries.
- [ ] `count_hints` returns 0 for an empty store.

## Implementation Notes

- **Primary file:** `src/hints.rs`
- **Dependencies:** HNT-003 (bidirectional indices provide the scan targets)
- **Performance:**
  - `get_coin_ids_by_hints` batch query should short-circuit once `max_items` is reached.
  - `count_hints` should use a materialized counter (incremented on insert, decremented on delete) rather than scanning the entire index.
- **Edge cases:**
  - `max_items = 0`: return empty result immediately.
  - Empty `hints` or `coin_ids` slice: return empty result immediately.
  - Very large result sets: `max_items` prevents unbounded memory allocation.
