# HNT-002: Idempotent Insertion

> **Authoritative requirement:** [HNT-002](../NORMATIVE.md#HNT-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

Hint insertion must be idempotent: inserting the same `(coin_id, hint)` pair more than once must silently succeed without creating duplicate entries or raising an error. This is critical because the same coin may appear in multiple block-processing contexts (e.g., re-processing after a reorg) and the hint store must remain consistent.

This directly mirrors Chia L1's `INSERT OR IGNORE` semantics in `hint_store.py`, which ensures that duplicate hint rows are silently discarded by the database.

---

## Specification

### Behavior

1. When `add_hint(coin_id, hint)` is called with a `(coin_id, hint)` pair that already exists in the store, the operation MUST be a silent no-op.
2. No error MUST be returned for duplicate insertions.
3. No duplicate entries MUST exist in either the forward or reverse index after any sequence of insertions.
4. The idempotency guarantee MUST hold regardless of insertion order or concurrency.

### Function Signature

```rust
impl HintStore {
    pub fn add_hint(&mut self, coin_id: &CoinId, hint: &[u8]) -> Result<(), HintError>;
}
```

### Semantics

- If the key `(coin_id, hint)` already exists in the forward index, skip insertion into both forward and reverse indices.
- A check-then-insert pattern or put-if-absent pattern MUST be used to avoid duplicates.

---

## Acceptance Criteria

- [ ] Inserting `(coin_id_A, hint_X)` twice returns `Ok(())` both times.
- [ ] After double insertion, querying hints for `coin_id_A` returns `hint_X` exactly once.
- [ ] After double insertion, querying coins for `hint_X` returns `coin_id_A` exactly once.
- [ ] Inserting the same coin_id with different hints creates distinct entries.
- [ ] Inserting different coin_ids with the same hint creates distinct entries.
- [ ] No panic or error on duplicate insertion under any ordering.

---

## Implementation Notes

- **Primary file:** `src/hints.rs`
- **Dependencies:** HNT-001 (validation must pass before insertion), HNT-003 (both indices must be updated atomically)
- **Pattern:** Use the KV store's native "put if absent" or check existence before insert. In redb, this can be done by checking if the key exists in the forward index before writing to both indices.
- **Edge cases:**
  - Bulk insertion of many hints for the same coin: each unique pair inserted once.
  - Re-processing the same block after rollback: all hints re-inserted idempotently.

## Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| `test_duplicate_hint_silent` | Unit | Insert same (coin_id, hint) twice | No error, single entry in store |
| `test_duplicate_forward_index` | Unit | Query hints for coin after double insert | Exactly one hint returned |
| `test_duplicate_reverse_index` | Unit | Query coins for hint after double insert | Exactly one coin_id returned |
| `test_same_coin_different_hints` | Unit | Insert (coin_A, hint_X) and (coin_A, hint_Y) | Two distinct entries |
| `test_different_coins_same_hint` | Unit | Insert (coin_A, hint_X) and (coin_B, hint_X) | Two distinct entries |
| `test_bulk_idempotent` | Unit | Insert 100 duplicate pairs in a batch | All silently ignored, single entries |
