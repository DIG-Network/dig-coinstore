# HNT-001: Hint Validation

> **Authoritative requirement:** [HNT-001](../NORMATIVE.md#HNT-001)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

Hints are optional byte sequences attached to coin creations that allow wallets to discover coins via puzzle-hash subscription. Not all hints are valid: empty hints carry no information and must be silently discarded, while oversized hints (exceeding 32 bytes) are malformed and must be rejected. Only exactly 32-byte hints are meaningful because they correspond to puzzle hashes used for subscription matching.

This mirrors Chia L1's `HintManager` validation logic, which skips zero-length hints and enforces a maximum hint length before storing.

---

## Specification

### Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `MAX_HINT_LENGTH` | `32` | Maximum allowed hint length in bytes. |

### Validation Rules

1. If `hint.len() == 0`, the hint MUST be silently skipped (no error, no storage).
2. If `hint.len() > MAX_HINT_LENGTH`, the hint MUST be rejected with a `HintTooLong` error.
3. If `1 <= hint.len() <= MAX_HINT_LENGTH`, the hint MUST be accepted for storage.
4. Only hints where `hint.len() == 32` are eligible for puzzle-hash subscription matching. Shorter valid hints MAY be stored but SHOULD NOT be used for subscription resolution.

### Error Type

```rust
pub enum HintError {
    HintTooLong { length: usize, max: usize },
}
```

### Validation Function

```rust
pub fn validate_hint(hint: &[u8]) -> Result<HintAction, HintError>;

pub enum HintAction {
    Skip,          // 0-byte hint, silently ignore
    Store,         // valid hint, store it
}
```

---

## Acceptance Criteria

- [ ] A 0-byte hint is silently skipped (returns `HintAction::Skip`, no error).
- [ ] A 33-byte hint is rejected with `HintError::HintTooLong { length: 33, max: 32 }`.
- [ ] A 32-byte hint is accepted (returns `HintAction::Store`).
- [ ] Hints between 1 and 31 bytes are accepted for storage.
- [ ] Only 32-byte hints participate in puzzle-hash subscription matching.
- [ ] `MAX_HINT_LENGTH` is defined as a public constant equal to `32`.

---

## Implementation Notes

- **Primary file:** `src/hints.rs`
- **Dependencies:** None (pure validation logic)
- **Edge cases:**
  - A hint of exactly `MAX_HINT_LENGTH` (32 bytes) is valid, not rejected.
  - Multiple hints per coin creation are each validated independently.
  - Validation occurs before any storage attempt (fail-fast).
