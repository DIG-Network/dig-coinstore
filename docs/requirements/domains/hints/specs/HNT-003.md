# HNT-003: Bidirectional Indices

> **Authoritative requirement:** [HNT-003](../NORMATIVE.md#HNT-003)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

The hint store maintains two KV indices to support efficient lookups in both directions: from a coin to its hints (forward) and from a hint to associated coins (reverse). Both indices use composite keys composed by concatenating the two 32-byte identifiers, with empty values (`()`) since the key itself encodes the full relationship.

This is an improvement over Chia L1's single SQL table with separate indexed columns. The KV composite-key approach enables prefix-scan queries for both directions without requiring secondary index maintenance.

---

## Specification

### Index Definitions

| Index | Key Format | Value | Purpose |
|-------|-----------|-------|---------|
| Forward | `coin_id (32 bytes) ++ hint (32 bytes)` | `()` | "Get all hints for a given coin" |
| Reverse | `hint (32 bytes) ++ coin_id (32 bytes)` | `()` | "Get all coins for a given hint" |

### Key Construction

```rust
// Forward key: coin_id || hint
fn forward_key(coin_id: &[u8; 32], hint: &[u8; 32]) -> [u8; 64] {
    let mut key = [0u8; 64];
    key[..32].copy_from_slice(coin_id);
    key[32..].copy_from_slice(hint);
    key
}

// Reverse key: hint || coin_id
fn reverse_key(hint: &[u8; 32], coin_id: &[u8; 32]) -> [u8; 64] {
    let mut key = [0u8; 64];
    key[..32].copy_from_slice(hint);
    key[32..].copy_from_slice(coin_id);
    key
}
```

### Lookup Patterns

1. **Get hints for coin:** Prefix-scan the forward index with `coin_id` as the 32-byte prefix. Each matching key's trailing 32 bytes is a hint.
2. **Get coins for hint:** Prefix-scan the reverse index with `hint` as the 32-byte prefix. Each matching key's trailing 32 bytes is a coin_id.

### Consistency Invariant

For every entry `(coin_id, hint)` in the forward index, there MUST exist a corresponding entry `(hint, coin_id)` in the reverse index, and vice versa. The two indices MUST always be in sync.

---

## Acceptance Criteria

- [ ] Forward index exists with composite key `coin_id ++ hint` and empty value.
- [ ] Reverse index exists with composite key `hint ++ coin_id` and empty value.
- [ ] Inserting a hint writes to both forward and reverse indices atomically.
- [ ] Deleting a hint removes from both forward and reverse indices atomically.
- [ ] Prefix scan on forward index with a coin_id returns all hints for that coin.
- [ ] Prefix scan on reverse index with a hint returns all coin_ids for that hint.
- [ ] No entry exists in one index without a corresponding entry in the other.

---

## Implementation Notes

- **Primary file:** `src/hints.rs`
- **Storage backend:** redb tables or equivalent KV store with ordered keys supporting prefix scans.
- **Dependencies:** STO-001 (KV storage layer)
- **Atomicity:** Both indices MUST be updated within the same write transaction to prevent inconsistency.
- **Edge cases:**
  - Coins with zero hints: no entries in either index (forward prefix scan returns empty).
  - Hints associated with thousands of coins: reverse prefix scan returns all, bounded by `max_items` at query time (HNT-004).
  - Non-32-byte hints (1-31 bytes): for forward index, key length varies. Consider padding or a separate table for short hints.

|------|------|-------------|-----------------|
| `test_forward_index_insert` | Unit | Insert hint, scan forward index by coin_id | Hint found in trailing bytes |
| `test_reverse_index_insert` | Unit | Insert hint, scan reverse index by hint | Coin ID found in trailing bytes |

| `test_delete_cleans_both` | Unit | Insert then delete a hint, scan both indices | Both indices empty for that pair |
| `test_atomic_update` | Integration | Insert with simulated failure mid-write | Either both or neither index updated |
| `test_short_hint_storage` | Unit | Insert a 16-byte hint, verify stored in both indices with correct key encoding | Forward and reverse keys handle variable length |
| `test_forward_reverse_consistency_invariant` | Integration | Insert 100 random hints, verify every forward entry has a matching reverse entry and vice versa | Counts match, no orphans in either direction |
