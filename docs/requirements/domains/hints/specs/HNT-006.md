# HNT-006: Variable-Length Hint Keys

> **Authoritative requirement:** [HNT-006](../NORMATIVE.md#HNT-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

The hint store must support variable-length hints in the range of 1-32 bytes. While 32-byte hints (corresponding to puzzle hashes) are the most common and semantically meaningful, shorter hints are valid per HNT-001 and must be stored correctly.

The key challenge is that the bidirectional index design (HNT-003) uses composite keys formed by concatenating `coin_id` and `hint`. When hints are shorter than 32 bytes, naive concatenation creates ambiguity: a prefix scan for one short hint could match entries for a different hint that shares the same prefix. To prevent this, non-32-byte hints MUST use padded or length-prefixed composite keys.

Additionally, only exactly 32-byte hints MUST participate in `include_hinted` joins in `batch_coin_states_by_puzzle_hashes()`. Shorter hints, while stored, are not eligible for puzzle-hash subscription matching because they do not represent valid puzzle hashes.

## Specification

### Key Encoding for Variable-Length Hints

Two approaches are acceptable:

#### Option A: Length-Prefixed Keys

```rust
// Forward key: coin_id (32 bytes) || hint_length (1 byte) || hint (1-32 bytes)
fn forward_key(coin_id: &[u8; 32], hint: &[u8]) -> Vec<u8> {
    let mut key = Vec::with_capacity(32 + 1 + hint.len());
    key.extend_from_slice(coin_id);
    key.push(hint.len() as u8);
    key.extend_from_slice(hint);
    key
}

// Reverse key: hint_length (1 byte) || hint (1-32 bytes) || coin_id (32 bytes)
fn reverse_key(hint: &[u8], coin_id: &[u8; 32]) -> Vec<u8> {
    let mut key = Vec::with_capacity(1 + hint.len() + 32);
    key.push(hint.len() as u8);
    key.extend_from_slice(hint);
    key.extend_from_slice(coin_id);
    key
}
```

#### Option B: Zero-Padded Keys

```rust
// Forward key: coin_id (32 bytes) || hint_padded (32 bytes, zero-filled)
fn forward_key(coin_id: &[u8; 32], hint: &[u8]) -> [u8; 64] {
    let mut key = [0u8; 64];
    key[..32].copy_from_slice(coin_id);
    key[32..32 + hint.len()].copy_from_slice(hint);
    // remaining bytes are 0
    key
}
```

**Recommendation:** Option A (length-prefixed) is preferred because it avoids collisions between a short hint and a longer hint that starts with the same bytes followed by zeros.

### Prefix-Scan Collision Prevention

Without length-prefixing or padding, the following collision scenario exists:
- Hint A: `[0x01, 0x02]` (2 bytes)
- Hint B: `[0x01, 0x02, 0x03]` (3 bytes)

A prefix scan for Hint A in the reverse index would also match entries for Hint B. Length-prefixing prevents this because:
- Hint A key starts with `[0x02, 0x01, 0x02, ...]` (length=2, then data)
- Hint B key starts with `[0x03, 0x01, 0x02, 0x03, ...]` (length=3, then data)

These have different prefixes and will not collide during prefix scans.

### Subscription Matching Restriction

Only 32-byte hints participate in the `include_hinted` join used by `batch_coin_states_by_puzzle_hashes()`:

```rust
fn batch_coin_states_by_puzzle_hashes(
    &self,
    puzzle_hashes: &[[u8; 32]],
    include_hinted: bool,
) -> Result<Vec<CoinState>, CoinStoreError> {
    // ... direct puzzle_hash lookup ...

    if include_hinted {
        // Only 32-byte hints are eligible for subscription matching
        for ph in puzzle_hashes {
            let hinted_coins = self.hint_store.get_coin_ids_by_hint(ph, max_items)?;
            // ... merge hinted coins into results ...
        }
    }
}
```

Shorter hints are stored for informational purposes (e.g., `get_hints_for_coin_ids()`) but do not affect subscription resolution.

## Acceptance Criteria

- [ ] Hints of length 1-31 bytes are stored correctly in both forward and reverse indices.
- [ ] Hints of length 32 bytes are stored correctly (unchanged from HNT-003).
- [ ] No prefix-scan collisions occur between hints of different lengths.
- [ ] No prefix-scan collisions occur between hints that share a common prefix but differ in length.
- [ ] `get_coin_ids_by_hint()` returns correct results for both short and 32-byte hints.
- [ ] `get_hints_for_coin_ids()` returns correct results including short hints.
- [ ] Only 32-byte hints participate in `include_hinted` joins.
- [ ] Short hints do NOT appear in `batch_coin_states_by_puzzle_hashes()` results when `include_hinted = true`.
- [ ] `count_hints()` counts both short and 32-byte hints.

## Implementation Notes

- **Primary file:** `src/hints.rs`
- **Dependencies:** HNT-001 (validation accepts 1-32 byte hints), HNT-003 (bidirectional index key format)
- **Migration:** If existing data used fixed 64-byte keys, a migration may be needed. For new implementations, use the variable-length format from the start.
- **Edge cases:**
  - 1-byte hint: smallest valid hint, must not collide with any other.
  - 31-byte hint: almost full-length, must not collide with 32-byte hints.
  - Hint that is all zeros: `[0x00]` (1 byte) vs `[0x00, 0x00]` (2 bytes) -- must be distinguishable.
  - Empty hint (0 bytes): already filtered by HNT-001, never reaches key construction.
