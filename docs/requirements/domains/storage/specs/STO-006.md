# STO-006: Compaction Strategy

> **Authoritative requirement:** [STO-006](../NORMATIVE.md#STO-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

Each RocksDB column family is configured with a compaction strategy matched to its access pattern. Read-optimized column families use Level compaction to minimize read amplification and maintain sorted data across levels. Append-mostly column families use FIFO compaction to avoid rewriting data that is rarely updated or read. Per-CF write buffer allocation ensures that high-throughput CFs have sufficient memtable capacity without starving low-throughput CFs.

---

## Specification

### Compaction Strategy Assignment

| Column Family | Strategy | Rationale |
|---------------|----------|-----------|
| `coin_records` | Level | Frequent point lookups; read amplification must be low |
| `coin_by_puzzle_hash` | Level | Frequent prefix scans; sorted order critical |
| `unspent_by_puzzle_hash` | Level | Frequent prefix scans; high churn from spend/unspend |
| `coin_by_parent` | Level | Prefix scans by parent; moderate read frequency |
| `coin_by_confirmed_height` | Level | Range scans by height; sorted order critical |
| `coin_by_spent_height` | Level | Range scans by height; sorted order critical |
| `hints` | Level | Point lookups and prefix scans |
| `hints_by_value` | Level | Prefix scans by hint value |
| `merkle_nodes` | Level | Frequent point lookups during proof generation |
| `archive_coin_records` | FIFO | Append-mostly; archived coins rarely read or updated |
| `state_snapshots` | FIFO | Append-mostly; old snapshots aged out naturally |
| `metadata` | Level | Small dataset; frequent reads of chain tip |

### Level Compaction Configuration

For Level-compacted CFs:

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `max_bytes_for_level_base` | 256 MiB | Base level size before triggering compaction |
| `max_bytes_for_level_multiplier` | 10 | Each level is 10x the previous |
| `level0_file_num_compaction_trigger` | 4 | Trigger compaction after 4 L0 files |
| `level0_slowdown_writes_trigger` | 20 | Slow down writes at 20 L0 files |
| `level0_stop_writes_trigger` | 36 | Stop writes at 36 L0 files |
| `target_file_size_base` | 64 MiB | SST file target size |
| `num_levels` | 7 | Default; sufficient for multi-TB datasets |

### FIFO Compaction Configuration

For FIFO-compacted CFs:

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `max_table_files_size` | 1 GiB (configurable) | Total size limit before oldest files are dropped |
| `allow_compaction` | false | Pure FIFO; no rewriting of files |

FIFO compaction drops the oldest SST files when the total size exceeds `max_table_files_size`. This is appropriate for append-only CFs where old data has diminishing value.

### Per-CF Write Buffer Allocation

| Column Family | `write_buffer_size` | `max_write_buffer_number` | Rationale |
|---------------|---------------------|--------------------------|-----------|
| `coin_records` | 64 MiB | 3 | High write throughput during sync |
| `coin_by_puzzle_hash` | 32 MiB | 2 | Moderate write volume |
| `unspent_by_puzzle_hash` | 32 MiB | 2 | Moderate write volume, high churn |
| `coin_by_parent` | 16 MiB | 2 | Lower write volume |
| `coin_by_confirmed_height` | 16 MiB | 2 | One entry per coin per block |
| `coin_by_spent_height` | 16 MiB | 2 | One entry per spend per block |
| `hints` | 16 MiB | 2 | Lower write volume |
| `hints_by_value` | 16 MiB | 2 | Lower write volume |
| `merkle_nodes` | 64 MiB | 3 | High write throughput during tree updates |
| `archive_coin_records` | 16 MiB | 2 | Infrequent writes |
| `state_snapshots` | 8 MiB | 2 | Infrequent writes |
| `metadata` | 4 MiB | 2 | Very small dataset |

Total write buffer memory: approximately 300 MiB across all CFs.

### Manual Compaction

The `StorageBackend::compact` method (STO-001) triggers a manual compaction on a specified CF. This is used:

- After initial bulk sync to optimize read performance.
- After a bloom filter configuration change (STO-004) to rebuild filters in compacted SSTs.
- On operator request for maintenance.

## Acceptance Criteria

- [x] Read-optimized CFs use Level compaction.
- [x] Append-mostly CFs (`archive_coin_records`, `state_snapshots`) use FIFO compaction.
- [x] Level compaction parameters match the specified configuration table.
- [x] FIFO compaction respects the `max_table_files_size` limit.
- [x] Each CF has its own `write_buffer_size` and `max_write_buffer_number`.
- [x] Manual compaction via `StorageBackend::compact` triggers compaction on the specified CF.
- [ ] Write stalls do not occur under normal sync load (L0 files stay below slowdown trigger).
- [x] LMDB backend (STO-003) `compact` is a no-op; this spec is RocksDB-specific.

## Implementation Notes

- **Primary file:** `src/storage/rocksdb.rs` (CF options builder)
- **Dependencies:** STO-002 (column families), STO-004 (bloom filter interplay)
- **Edge cases:**
  - FIFO compaction with TTL: not currently used, but could be added for time-based expiry of archive data.
  - FIFO compaction does not support range deletes well; rollback of height indices may need special handling.
  - Write stalls during initial sync: monitor L0 file count; consider temporarily raising `level0_slowdown_writes_trigger` during bulk sync.
  - LMDB does not have configurable compaction; STO-003 backend's `compact` method is a no-op.
  - Total write buffer memory: sum of all CF write buffers is approximately 300 MiB. Ensure the host has sufficient memory.
