# STO-003: LMDB Backend

## Summary

The LMDB backend provides a lightweight, memory-mapped alternative to RocksDB for environments where simplicity and predictable latency are preferred over tunable throughput. It uses 6 named databases within a single LMDB environment. LMDB's copy-on-write B+ tree design provides MVCC (multi-version concurrency control) with lock-free reads and serialized writes, making it well-suited for read-heavy workloads with moderate write volume.

## Specification

### Named Databases

| Database | Purpose | Key Format | Value Format |
|----------|---------|------------|--------------|
| `coins` | Primary coin record storage | coin_id (32 bytes) | bincode CoinRecord |
| `coins_by_ph` | Coins indexed by puzzle hash | puzzle_hash + coin_id | empty |
| `hints` | Forward hint index | coin_id + hint | empty |
| `hints_by_value` | Reverse hint index | hint + coin_id | empty |
| `snapshots` | Checkpoint snapshots | height (u32 BE) | bincode snapshot |
| `metadata` | Chain state metadata | string key | variable |

### Initialization

On open, the LMDB environment MUST be configured with:

- `max_dbs` set to at least 6 to accommodate all named databases.
- `map_size` set to a configurable value (default: 1 GiB) to define the memory-mapped region.
- All 6 named databases MUST be opened (created if they do not exist) within a single write transaction using `DatabaseFlags::CREATE`.

### Memory-Mapped I/O

LMDB relies on the operating system's virtual memory subsystem for caching. No application-level cache is required or used. Pages are faulted in on demand and evicted by the OS page cache. This eliminates double-caching and provides predictable memory behavior.

### MVCC Semantics

- Readers acquire a read transaction that provides a consistent snapshot of the database at the time the transaction was opened.
- A single write transaction may be active at any time; concurrent write attempts block until the current writer commits or aborts.
- Read transactions do not block writers and writers do not block readers.
- Stale read transactions prevent page reclamation -- long-lived readers can cause the database file to grow.

### Environment Configuration

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `max_dbs` | 6 | One per named database |
| `map_size` | 1 GiB (configurable) | Room for growth; resizable at runtime on most platforms |
| `max_readers` | 126 (default) | Sufficient for concurrent read threads |
| `NO_TLS` | enabled | Allows read transactions to be used across threads |
| `NO_READAHEAD` | enabled | Avoids unnecessary readahead for random access patterns |

### Database Mapping to RocksDB Column Families

LMDB uses fewer databases than RocksDB column families because LMDB's B+ tree handles some access patterns natively:

| LMDB Database | Equivalent RocksDB CFs |
|---------------|------------------------|
| `coins` | `coin_records`, `coin_by_parent`, `coin_by_confirmed_height`, `coin_by_spent_height` |
| `coins_by_ph` | `coin_by_puzzle_hash`, `unspent_by_puzzle_hash` |
| `hints` | `hints` |
| `hints_by_value` | `hints_by_value` |
| `snapshots` | `state_snapshots` |
| `metadata` | `metadata` |

Note: `merkle_nodes` and `archive_coin_records` from RocksDB are folded into `coins` or omitted in the LMDB backend depending on configuration.

## Acceptance Criteria

- [x] All 6 named databases are created on environment open.
- [x] The backend implements the `StorageBackend` trait (STO-001).
- [x] Read transactions provide a consistent snapshot (MVCC).
- [x] Concurrent readers do not block writes and vice versa.
- [x] `map_size` is configurable at initialization.
- [x] Data persists across environment close and reopen.
- [x] `prefix_scan` works correctly using LMDB cursor with `MDB_SET_RANGE`.

## Implementation Notes

- **Primary file:** `src/storage/lmdb.rs`
- **Dependencies:** STO-001 (trait), STO-007 (feature gate `lmdb-storage`), STO-008 (serialization)
- **Edge cases:**
  - Map full (`MDB_MAP_FULL`): return `StorageError::MapFull` and advise increasing `map_size`. The caller should resize the environment and retry.
  - Stale readers: long-lived read transactions can prevent reclamation of old pages, causing unbounded file growth. Consider read transaction timeouts or monitoring via `mdb_reader_check`.
  - `prefix_scan` implementation: LMDB cursors support `MDB_SET_RANGE`; position cursor at the first key >= prefix, then iterate forward until the key prefix no longer matches.
  - Maximum readers exceeded: return `StorageError` with clear message; operators may need to increase `max_readers`.
  - Platform note: LMDB memory-mapping behavior may differ between Linux and macOS. Windows support requires careful handling of `map_size` resizing (environment must be closed and reopened).
  - LMDB does not support column-family-level bloom filters; point lookup performance depends on B+ tree depth. For typical coin-state databases this is acceptable.
