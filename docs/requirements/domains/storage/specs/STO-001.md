# STO-001: StorageBackend Trait

> **Authoritative requirement:** [STO-001](../NORMATIVE.md#STO-001)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

The `StorageBackend` trait defines the abstract interface for all persistent storage operations in the coin store. By programming against a trait rather than a concrete backend, the coin store can support multiple storage engines (RocksDB, LMDB) and enable testing with in-memory backends. The trait covers point operations (get/put/delete), batch writes, prefix scans, and maintenance operations (flush/compact).

---

## Specification

### Trait Definition

```rust
pub trait StorageBackend: Send + Sync {
    fn get(&self, cf: &str, key: &[u8]) -> Result<Option<Vec<u8>>, StorageError>;
    fn put(&self, cf: &str, key: &[u8], value: &[u8]) -> Result<(), StorageError>;
    fn delete(&self, cf: &str, key: &[u8]) -> Result<(), StorageError>;
    fn batch_write(&self, batch: WriteBatch) -> Result<(), StorageError>;
    fn prefix_scan(&self, cf: &str, prefix: &[u8]) -> Result<Vec<(Vec<u8>, Vec<u8>)>, StorageError>;
    fn flush(&self) -> Result<(), StorageError>;
    fn compact(&self, cf: &str) -> Result<(), StorageError>;
}
```

### Method Semantics

| Method | Description |
|--------|-------------|
| `get` | Retrieve a value by column family and key. Returns `None` if not found. |
| `put` | Insert or update a key-value pair in the specified column family. |
| `delete` | Remove a key from the specified column family. No error if key doesn't exist. |
| `batch_write` | Atomically apply all operations in a `WriteBatch`. |
| `prefix_scan` | Return all key-value pairs in a CF where the key starts with the given prefix. |
| `flush` | Force WAL and memtable flush to persistent storage. |
| `compact` | Trigger compaction on the specified column family. |

### Thread Safety

The trait requires `Send + Sync` to support concurrent access from multiple threads (CON-001).

---

## Acceptance Criteria

- [x] The `StorageBackend` trait is defined with all 7 methods.
- [x] The trait requires `Send + Sync`.
- [x] Both RocksDB (STO-002) and LMDB (STO-003) backends implement the trait.
- [x] `get` returns `None` for missing keys (not an error).
- [x] `delete` is idempotent (no error for missing keys).
- [x] `batch_write` is atomic (all or nothing).

---

## Implementation Notes

- **Primary file:** `src/storage/mod.rs` or `src/storage/backend.rs`
- **Dependencies:** CON-001 (Send + Sync)
- **Edge cases:**
  - Invalid column family name: return `StorageError`.
  - Empty `WriteBatch`: no-op, no error.
  - `prefix_scan` with empty prefix: returns all entries in the CF (use with caution).
