# STO-002: RocksDB Backend

> **Authoritative requirement:** [STO-002](../NORMATIVE.md#STO-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The RocksDB backend is the primary storage engine for the coin store. It uses 12 column families to segregate different data types for optimal access patterns, compaction strategies, and bloom filter configurations. Each column family serves a specific query or storage purpose, enabling independent tuning of bloom filters (STO-004), compaction (STO-006), and write buffer allocation per data category.

---

## Specification

### Column Families

| Column Family | Purpose | Key Format | Value Format |
|---------------|---------|------------|--------------|
| `coin_records` | Primary coin record storage | coin_id (32 bytes) | bincode CoinRecord |
| `coin_by_puzzle_hash` | All coins by puzzle hash | puzzle_hash + coin_id (64 bytes) | empty |
| `unspent_by_puzzle_hash` | Unspent coins by puzzle hash | puzzle_hash + coin_id (64 bytes) | empty |
| `coin_by_parent` | Coins by parent coin ID | parent_id + coin_id (64 bytes) | empty |
| `coin_by_confirmed_height` | Coins by confirmed height | height (8B u64 BE) + coin_id (40 bytes) | empty |
| `coin_by_spent_height` | Coins by spent height | height (8B u64 BE) + coin_id (40 bytes) | empty |
| `hints` | Forward hint index | coin_id + hint (64 bytes) | empty |
| `hints_by_value` | Reverse hint index | hint + coin_id (64 bytes) | empty |
| `merkle_nodes` | Sparse Merkle tree nodes | node_path (variable) | Bytes32 hash |
| `archive_coin_records` | Archived spent coins | coin_id (32 bytes) | bincode CoinRecord |
| `state_snapshots` | Checkpoint snapshots | height (8B u64 BE) | bincode snapshot |
| `metadata` | Chain state metadata | string key | variable |

### Per-CF Configuration Summary

Each column family has specific bloom filter and compaction configuration:

| Column Family | Bloom Filter | Compaction | Write Buffer |
|---------------|-------------|------------|--------------|
| `coin_records` | Full (10 bits/key) | Level | 64 MiB |
| `coin_by_puzzle_hash` | Prefix (32B) | Level | 32 MiB |
| `unspent_by_puzzle_hash` | Prefix (32B) | Level | 32 MiB |
| `coin_by_parent` | Full (10 bits/key) | Level | 16 MiB |
| `coin_by_confirmed_height` | None | Level | 16 MiB |
| `coin_by_spent_height` | None | Level | 16 MiB |
| `hints` | Full (10 bits/key) | Level | 16 MiB |
| `hints_by_value` | Prefix (32B) | Level | 16 MiB |
| `merkle_nodes` | Full (10 bits/key) | Level | 64 MiB |
| `archive_coin_records` | Full (10 bits/key) | Level | 16 MiB |
| `state_snapshots` | None | FIFO | 8 MiB |
| `metadata` | Full (10 bits/key) | Level | 4 MiB |

See STO-004 for bloom filter details, STO-006 for compaction strategy details.

### Initialization

On open, all 12 column families MUST be created if they do not exist. RocksDB MUST be configured with:

- `create_if_missing(true)` -- create the database directory if it does not exist.
- `create_missing_column_families(true)` -- create any CFs that are missing (supports schema evolution).
- Per-CF `Options` with bloom filter (STO-004), compaction (STO-006), and write buffer settings applied.
- `set_keep_log_file_num(10)` -- retain the last 10 LOG files for debugging.

### Global RocksDB Options

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `max_background_jobs` | 4 | Concurrent compaction and flush threads |
| `bytes_per_sync` | 1 MiB | Periodic fsync during compaction to limit OS page cache pressure |
| `wal_bytes_per_sync` | 1 MiB | Periodic WAL fsync for durability |
| `max_total_wal_size` | 256 MiB | Limit total WAL size to trigger CF flush |

## Acceptance Criteria

- [ ] All 12 column families are created on database open.
- [ ] Each CF has appropriate bloom filter configuration (STO-004).
- [ ] Each CF has appropriate compaction strategy (STO-006).
- [ ] Each CF has appropriate write buffer allocation (STO-006).
- [ ] Data is correctly isolated between column families.
- [ ] The backend implements the `StorageBackend` trait (STO-001).
- [ ] Reopening an existing database preserves all data and CFs.
- [ ] Missing CFs are created on reopen (schema evolution support).

## Implementation Notes

- **Primary file:** `src/storage/rocksdb.rs`
- **Dependencies:** STO-001 (trait), STO-004 (bloom), STO-006 (compaction), STO-007 (feature gate `rocksdb-storage`), STO-008 (serialization)
- **Edge cases:**
  - Opening an existing database with a new CF: RocksDB handles this via `create_missing_column_families(true)`.
  - Concurrent reads across CFs: RocksDB supports this natively; each CF has independent memtables and SST files.
  - Corrupted database: RocksDB provides `repair_db` as a recovery mechanism. Consider exposing this via a CLI command.
  - Large number of CFs (12): each CF maintains its own memtable and flush pipeline. Monitor total memory usage (approximately 300 MiB for write buffers alone).
