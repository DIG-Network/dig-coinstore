# STO-004: Bloom Filter Configuration

> **Authoritative requirement:** [STO-004](../NORMATIVE.md#STO-004)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

Bloom filters are configured per column family in the RocksDB backend to minimize unnecessary disk reads. Point-lookup column families use full bloom filters (10 bits/key) to quickly reject absent keys with an approximate 1% false positive rate. Puzzle-hash column families use prefix bloom filters (32-byte prefix) to accelerate prefix scans. Sequential-access column families disable bloom filters entirely since their access patterns (range iteration) do not benefit from probabilistic membership tests. L0 filter and index blocks are pinned in the block cache to avoid repeated I/O for the most recently flushed data.

## Specification

### Bloom Filter Categories

| Category | Filter Type | Configuration | Column Families |
|----------|-------------|---------------|-----------------|
| Point lookup | Full bloom | 10 bits/key | `coin_records`, `coin_by_parent`, `hints`, `archive_coin_records`, `merkle_nodes`, `metadata` |
| Prefix scan | Prefix bloom | 32-byte prefix | `coin_by_puzzle_hash`, `unspent_by_puzzle_hash`, `hints_by_value` |
| Sequential | None | Bloom disabled | `coin_by_confirmed_height`, `coin_by_spent_height`, `state_snapshots` |

### Full Bloom Filters (10 bits/key)

Full bloom filters are created using `BlockBasedTableOptions::set_bloom_filter(10, false)`. The `false` parameter indicates a full (non-block-based) filter. At 10 bits/key, the false positive rate is approximately 1%.

- Applied to CFs where the primary access pattern is `get(key)`.
- The filter is checked before reading any SST data block, avoiding disk I/O for absent keys.

### Prefix Bloom Filters (32 bytes)

Prefix bloom filters are created by combining a prefix extractor with a bloom filter:

```rust
opts.set_prefix_extractor(SliceTransform::create_fixed_prefix(32));
block_opts.set_bloom_filter(10, false);
```

- Applied to CFs where the primary access pattern is `prefix_scan(puzzle_hash)`.
- The 32-byte prefix corresponds to the puzzle hash length (Bytes32).
- Enables efficient `seek` and `seek_for_prev` operations on the prefix.
- `set_memtable_prefix_bloom_ratio(0.1)` SHOULD be set to enable memtable bloom for prefix lookups.

### No Bloom Filter

Sequential-access CFs do not configure any bloom filter. These CFs are accessed by range scans (e.g., all coins confirmed at height X) where bloom filters provide no benefit and would waste memory.

### L0 Filter and Index Pinning

For all CFs with bloom filters enabled:

```rust
block_opts.set_pin_l0_filter_and_index_blocks_in_cache(true);
```

This pins the filter and index blocks of L0 SST files in the block cache, preventing eviction. L0 files are the most recently flushed and most likely to be accessed during normal operation.

| Column Family | Bloom Type | Bits/Key | Prefix Length | Pin L0 |

## Acceptance Criteria

- [x] Point-lookup CFs have full bloom filters configured at 10 bits/key.
- [x] Prefix-scan CFs have prefix bloom filters with a 32-byte prefix extractor.
- [x] Sequential CFs have no bloom filter configured.
- [x] L0 filter and index blocks are pinned in cache for all bloom-enabled CFs.
- [x] False positive rate for full bloom is approximately 1% (10 bits/key).
- [x] Bloom filter configuration is applied at column family open time.
- [x] LMDB backend (STO-003) is unaffected; this spec is RocksDB-specific.

## Implementation Notes

- **Primary file:** `src/storage/rocksdb.rs` (CF options builder)
- **Dependencies:** STO-002 (column families), STO-006 (compaction interplay)
- **Edge cases:**
  - Prefix bloom with keys shorter than 32 bytes: RocksDB uses the full key as the prefix. Ensure all puzzle-hash CF keys are at least 32 bytes.
  - Changing bloom configuration on an existing database: existing SST files retain their original bloom filters until compacted. A manual compaction (STO-006) may be needed after configuration changes.
  - Memory impact: bloom filters consume approximately 10 bits per key. For 100M coins, this is roughly 120 MB of filter data across all CFs.
  - Memtable prefix bloom: `set_memtable_prefix_bloom_ratio` should be set for prefix CFs to enable bloom filtering on the memtable before flush.
