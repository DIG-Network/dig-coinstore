# STO-007: Feature Gates

> **Authoritative requirement:** [STO-007](../NORMATIVE.md#STO-007)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

Storage backends are gated behind Cargo feature flags to allow compile-time selection of the storage engine. Three feature flags control which backends are compiled: `rocksdb-storage`, `lmdb-storage`, and `full-storage` (which enables both). The default feature is `rocksdb-storage`. Conditional compilation via `#[cfg(feature = "...")]` ensures that unused backend code and its native dependencies are excluded from the binary, reducing compile time and binary size.

---

## Specification

### Feature Flag Definitions

In `Cargo.toml`:

```toml
[features]
default = ["rocksdb-storage"]
rocksdb-storage = ["dep:rocksdb"]
lmdb-storage = ["dep:heed"]
full-storage = ["rocksdb-storage", "lmdb-storage"]
```

### Feature Matrix

| Feature | RocksDB Backend | LMDB Backend | Default |
|---------|----------------|--------------|---------|
| `rocksdb-storage` | Compiled | Excluded | Yes |
| `lmdb-storage` | Excluded | Compiled | No |
| `full-storage` | Compiled | Compiled | No |
| (none) | Excluded | Excluded | N/A (compile error) |

### Conditional Compilation

Backend modules MUST be conditionally compiled:

```rust
#[cfg(feature = "rocksdb-storage")]
pub mod rocksdb;

#[cfg(feature = "lmdb-storage")]
pub mod lmdb;
```

### Compile-Time Validation

If no storage feature is enabled, compilation MUST fail with a clear error message (implemented in `src/lib.rs` next to the crate re-exports):

```rust
#[cfg(not(any(feature = "rocksdb-storage", feature = "lmdb-storage")))]
compile_error!(
    "dig-coinstore: enable at least one storage backend Cargo feature: \
     `rocksdb-storage` and/or `lmdb-storage`, or `full-storage` for both. \
     The default package features include `rocksdb-storage`."
);
```

### Backend Selection at Runtime

When multiple backends are compiled (via `full-storage`), the backend is selected at runtime via `config::StorageBackend` and `storage::open_storage_backend` (see `src/storage/mod.rs`, re-exported from `src/lib.rs` as `open_storage_backend`).

### Dependency Isolation

- The `rocksdb` crate (C++ dependency, long compile time) is only pulled in when `rocksdb-storage` is enabled.
- The `heed` crate (LMDB bindings) is only pulled in when `lmdb-storage` is enabled.
- Shared types (`WriteBatch`, `StorageError`, `StorageBackend` trait) are always compiled regardless of feature flags.

### Conditional Test Modules

Tests specific to a backend MUST also be gated behind the corresponding feature flag:

```rust
#[cfg(test)]
#[cfg(feature = "rocksdb-storage")]
mod rocksdb_tests { ... }

#[cfg(test)]
#[cfg(feature = "lmdb-storage")]
mod lmdb_tests { ... }
```

## Acceptance Criteria

- [x] `rocksdb-storage` is the default feature.
- [x] `cargo build` with default features compiles only the RocksDB backend.
- [x] `cargo build --features lmdb-storage --no-default-features` compiles only the LMDB backend.
- [x] `cargo build --features full-storage` compiles both backends.
- [x] `cargo build --no-default-features` fails with a `compile_error!` message.
- [x] RocksDB C++ dependencies are not linked when only `lmdb-storage` is enabled (optional `dep:rocksdb` off).
- [x] LMDB (`heed`) is not linked when only `rocksdb-storage` is enabled (optional `dep:heed` off).
- [x] The `StorageBackend` trait and shared types compile under all valid feature combinations.
- [x] Backend-specific tests are gated behind their respective feature flags (`tests/sto_007_tests.rs` and existing STO suites).

## Implementation Notes

- **Primary files:** `Cargo.toml`, `src/storage/mod.rs`
- **Dependencies:** STO-002 (`rocksdb-storage`), STO-003 (`lmdb-storage`), STO-001 (trait always available)
- **Edge cases:**
  - CI matrix: tests should run with each feature flag independently and with `full-storage` to ensure no cross-contamination between features.
  - Feature flag typos: Cargo silently ignores unknown features. Use `cargo clippy` and CI checks to catch issues.
  - Conditional test modules: tests specific to a backend should also be gated behind the corresponding feature flag to avoid compilation errors.
  - `--no-default-features` with no explicit feature: produces a compile error via `compile_error!` macro.
