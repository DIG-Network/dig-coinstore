# STO-008: Serialization

> **Authoritative requirement:** [STO-008](../NORMATIVE.md#STO-008)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

All structured data persisted to the storage backend is serialized using bincode for compact, deterministic binary encoding. `CoinRecord` and snapshot structures use bincode for value serialization. Key encoding helpers in `schema.rs` provide functions for constructing composite keys (e.g., puzzle_hash + coin_id) used by secondary index column families. Key encoding is fixed-width and big-endian to preserve lexicographic sort order in the storage engine.

**Dual serialization strategy:** `bincode` is used for internal KV storage (more compact, no CLVM overhead). `chia-protocol` types like `CoinState` and `Coin` natively support `chia-traits::Streamable` serialization for wire-format interchange (sync protocol, peer messages). The crate does NOT use `Streamable` for internal persistence — `bincode` with fixed-width big-endian encoding is more efficient for KV storage. Wire-format serialization of `CoinState` is the caller's responsibility using the `Streamable` trait from `chia-traits`.

## Specification

### Value Serialization (bincode)

The following types are serialized using `bincode::encode` / `bincode::decode`:

| Type | Used In | Serialized Fields |
|------|---------|-------------------|
| `CoinRecord` | `coin_records`, `archive_coin_records` (RocksDB); `coins` (LMDB) | coin, confirmed_height, spent_height, coinbase, timestamp |
| `CoinStoreSnapshot` (spec name `StateSnapshot`) | `state_snapshots` (RocksDB); `snapshots` (LMDB) | height, block_hash, state_root, coins, hints, totals, timestamp |

Bincode configuration (this crate pins **bincode 1.3**; the flags match the normative intent of newer `bincode` docs):

```rust
use bincode::Options;

let opts = bincode::DefaultOptions::new()
    .with_fixint_encoding()
    .with_big_endian();
```

- **Fixed-width integers:** ensures deterministic size regardless of value magnitude (no varint encoding).
- **Big-endian:** ensures serialized integers sort lexicographically the same as their numeric order.

### Key Encoding Helpers (schema.rs)

The `schema.rs` module provides helper functions for constructing and deconstructing composite keys:

```rust
// schema.rs

/// Construct a key for coin_records / archive_coin_records / coins
/// Key: coin_id (32 bytes)
pub fn coin_key(coin_id: &Bytes32) -> [u8; 32];

/// Construct a key for coin_by_puzzle_hash / unspent_by_puzzle_hash / coins_by_ph
/// Key: puzzle_hash (32 bytes) + coin_id (32 bytes) = 64 bytes
pub fn puzzle_hash_coin_key(puzzle_hash: &Bytes32, coin_id: &Bytes32) -> [u8; 64];

/// Extract puzzle_hash from a composite key
pub fn puzzle_hash_from_key(key: &[u8]) -> Bytes32;

/// Construct a key for coin_by_parent
/// Key: parent_id (32 bytes) + coin_id (32 bytes) = 64 bytes
pub fn parent_coin_key(parent_id: &Bytes32, coin_id: &Bytes32) -> [u8; 64];

/// Construct a key for coin_by_confirmed_height / coin_by_spent_height
/// Key: height (8 bytes, u64 BE) + coin_id (32 bytes) = 40 bytes
pub fn height_coin_key(height: u64, coin_id: &Bytes32) -> [u8; 40];

/// Extract height from a composite key
pub fn height_from_key(key: &[u8]) -> u64;

/// Construct a key for hints
/// Key: coin_id (32 bytes) + hint (32 bytes) = 64 bytes
pub fn coin_hint_key(coin_id: &Bytes32, hint: &Bytes32) -> [u8; 64];

/// Construct a key for hints_by_value
/// Key: hint (32 bytes) + coin_id (32 bytes) = 64 bytes
pub fn hint_coin_key(hint: &Bytes32, coin_id: &Bytes32) -> [u8; 64];

/// Construct a key for state_snapshots / snapshots
/// Key: height (8 bytes, u64 BE)
pub fn snapshot_key(height: u64) -> [u8; 8];

/// Construct a key for metadata
/// Key: string key as UTF-8 bytes (variable length)
pub fn metadata_key(name: &str) -> Vec<u8>;
```

### Composite Key Construction Rules

1. **Fixed width:** all key components are fixed-width (except `metadata`). No length prefixes or delimiters.
2. **Big-endian integers:** `u64` heights are encoded as 8-byte big-endian to preserve numeric sort order in lexicographic key comparison. The SPEC uses `u64` for heights throughout.
3. **Concatenation:** composite keys are constructed by concatenating components directly (e.g., `puzzle_hash || coin_id`).
4. **Prefix compatibility:** the first component of a composite key is always the prefix used for `prefix_scan` operations (STO-004 prefix bloom alignment).

## Acceptance Criteria

- [x] `CoinRecord` round-trips through bincode serialization without data loss.
- [x] `CoinStoreSnapshot` round-trips through bincode serialization without data loss.
- [x] Bincode uses fixed-width integer encoding and big-endian byte order.
- [x] All key encoding helpers in `schema.rs` produce fixed-width keys (except `metadata_key`).
- [x] Height-based keys sort lexicographically in ascending numeric order.
- [x] Puzzle-hash composite keys support prefix extraction of the first 32 bytes.
- [x] Key encoding/decoding functions are inverses (roundtrip property).
- [x] Bincode serialization is deterministic (same input always produces same bytes).

## Implementation Notes

- **Primary files:** `src/storage/kv_bincode.rs` (STO-008 bincode options + encode/decode + legacy fallbacks),
  `src/storage/schema.rs` (key helpers), `src/types.rs` (`CoinRecord` / `CoinStoreSnapshot` serde derives),
  `src/coin_store.rs` (snapshot persistence, FF coin rows, coin row decode order, hint keys on restore).
- **Merkle node keys:** internal SMT nodes use a fixed **33-byte** key `level (1B) || path (32B)` — see
  `schema::merkle_node_key` / `merkle_node_from_key` and MRK-003 (`src/merkle/persistent.rs`). Leaf / proof keys
  may remain variable-length where the Merkle module documents otherwise.
- **Dependencies:** STO-002 (RocksDB CF key/value formats), STO-003 (LMDB database key/value formats), STO-004 (prefix bloom alignment)
- **Edge cases:**
  - Bincode version compatibility: pinning the `bincode` crate version is critical. A major version change could break deserialization of existing data. Consider a version byte or migration strategy.
  - Zero-height keys: `height_coin_key(0, ...)` produces `[0, 0, 0, 0, 0, 0, 0, 0, ...]` (8 zero bytes), which is a valid key. Ensure no off-by-one issues with genesis block.
  - Variable-length keys (`merkle_nodes`, `metadata`): these CFs do not use the fixed-width key helpers. Key format is documented but not enforced by `schema.rs`.
  - Empty values: secondary index CFs store empty values `[]`. The key itself carries all information needed.
  - `Bytes32` alignment: key helpers accept `&Bytes32` references and copy bytes into the output array. No heap allocation for fixed-width keys.
