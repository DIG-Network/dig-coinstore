# STO-005: WriteBatch Atomic Commits

> **Authoritative requirement:** [STO-005](../NORMATIVE.md#STO-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

All writes for a single block MUST be committed atomically via a single `WriteBatch` with one WAL fsync. This guarantees that readers never observe a partially applied block -- either all mutations for a block are visible or none are. Batching writes also provides a 10-50x throughput improvement over individual `put` operations by amortizing the WAL sync cost across all operations in the batch. This is an improvement over Chia L1, which performs individual SQLite writes per coin operation.

---

## Specification

### WriteBatch Structure

A `WriteBatch` aggregates all mutations (puts, deletes) across multiple column families for a single block into one atomic unit:

```rust
pub struct WriteBatch {
    ops: Vec<WriteBatchOp>,
}

pub enum WriteBatchOp {
    Put { cf: String, key: Vec<u8>, value: Vec<u8> },
    Delete { cf: String, key: Vec<u8> },
}
```

### Block Processing Flow

For each new block, the coin store assembles a `WriteBatch` containing:

1. **Coin record inserts** -- new `CoinRecord` entries in `coin_records`.
2. **Index updates** -- secondary index entries in `coin_by_puzzle_hash`, `unspent_by_puzzle_hash`, `coin_by_parent`, `coin_by_confirmed_height`.
3. **Spend processing** -- mark coins as spent in `coin_records`, add `coin_by_spent_height` entries, remove from `unspent_by_puzzle_hash`.
4. **Hint inserts** -- hint entries in `hints` and `hints_by_value`.
5. **Merkle updates** -- updated nodes in `merkle_nodes`.
6. **Metadata update** -- update peak height and block hash in `metadata`.

All of these operations are added to a single `WriteBatch` before calling `batch_write`.

### Atomicity Guarantee

- **RocksDB:** `WriteBatch` is written to the WAL as a single record. `write_opt.set_sync(true)` ensures a single `fsync` for durability. On crash, either the entire batch is replayed from the WAL or none of it is.
- **LMDB:** All operations are performed within a single write transaction. `mdb_txn_commit` provides atomicity and durability.

### Performance Characteristics

| Approach | Throughput | WAL Syncs per Block |
|----------|-----------|---------------------|
| Individual `put` calls | Baseline (1x) | N (one per put) |
| `WriteBatch` | 10-50x faster | 1 |

The performance improvement comes from:
- A single WAL write and fsync instead of N.
- Reduced lock acquisition overhead.
- Batch memtable insertion.

### Ordering Within a Batch

Operations within a `WriteBatch` are applied in insertion order. The coin store MUST add operations in a deterministic order to ensure consistent state across nodes.

### Error Handling

- If `batch_write` fails, no operations from the batch are applied.
- The caller MUST NOT retry a partial batch; the entire batch should be reconstructed and retried.
- A failed batch MUST NOT leave the database in an inconsistent state.

### Guarantees Summary

| Property | Description |
|----------|-------------|
| **Atomicity** | All operations in the batch succeed or none do. |
| **Durability** | A single WAL fsync ensures the batch is durable on commit. |
| **Isolation** | Concurrent readers see either pre-batch or post-batch state, never partial. |
| **Crash safety** | On crash before commit, no changes visible. On crash after commit, all changes visible. |

## Acceptance Criteria

- [ ] All writes for a block are collected into a single `WriteBatch`. *(Orchestration lives in future BLK-* / `CoinStore::apply_block`; storage primitive is ready.)*
- [x] `batch_write` issues exactly one WAL fsync (RocksDB) or one `mdb_txn_commit` (LMDB).
- [x] Readers never observe a partially applied block.
- [x] A failed `batch_write` leaves the database unchanged.
- [x] Operations within the batch are applied in deterministic order.
- [x] Empty `WriteBatch` is a no-op (no fsync, no error).
- [x] Throughput improvement of at least 10x over individual puts for typical block sizes.

## Implementation Notes

- **Primary files:** `src/storage/mod.rs` (WriteBatch struct), `src/storage/rocksdb.rs`, `src/storage/lmdb.rs`
- **Dependencies:** STO-001 (trait `batch_write`), STO-002 (RocksDB CFs), STO-003 (LMDB databases)
- **Edge cases:**
  - Empty `WriteBatch`: calling `batch_write` with no operations is a no-op and MUST NOT error or fsync.
  - Very large batches (reorgs spanning many blocks): memory usage scales with batch size. Consider splitting reorg rollbacks into per-block batches.
  - Concurrent `batch_write` calls: RocksDB serializes WAL writes internally. LMDB allows only one write transaction at a time; concurrent attempts block.
  - Write failure mid-batch: RocksDB guarantees no partial writes via WAL. LMDB guarantees no partial writes via transaction abort on error.
