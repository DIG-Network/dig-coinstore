# RBK-002: Coin Deletion During Rollback

> **Authoritative requirement:** [RBK-002](../NORMATIVE.md#RBK-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

During rollback, coins that were confirmed (created) after the target height must be completely removed from the coin store. Before deletion, each coin's full record is saved to the `modified_coins` map in the `RollbackResult`, preserving a snapshot of the state that was rolled back. This enables callers (such as the mempool) to react to rolled-back coins.

This corresponds to the first phase of Chia L1's `rollback()` method, where coins with `confirmed_block_index > fork_height` are deleted from the database.

---

## Specification

### Behavior

1. Query all coins where `confirmed_height > target_height`.
2. For each such coin, save the full `CoinRecord` to `modified_coins` keyed by `coin_id`.
3. Delete the coin from all indices (primary record, puzzle hash index, parent index, height index, unspent index, Merkle tree).
4. Delete associated hints (see HNT-005).
5. Update materialized counters (decrement unspent_count/spent_count, subtract from total_value as appropriate).

### Ordering

Coin deletion MUST occur before coin un-spending (RBK-003) to avoid conflicts where a coin created in a rolled-back block was also spent in a later rolled-back block.

---

## Acceptance Criteria

- [ ] All coins with `confirmed_height > target_height` are deleted from the store.
- [ ] Pre-deletion `CoinRecord` snapshots are present in `modified_coins`.
- [ ] All secondary indices are cleaned up (puzzle hash, parent, height, unspent).
- [ ] Merkle tree entries for deleted coins are removed.
- [ ] Associated hints are deleted (cross-reference HNT-005).
- [ ] Materialized counters are decremented correctly.

---

## Implementation Notes

- **Primary file:** `src/rollback.rs`
- **Dependencies:** STO-002/STO-003 (storage indices), HNT-005 (hint cleanup), PRF-003 (counters), MRK-001 (Merkle removal)
- **Edge cases:**
  - Coins both created and spent in rolled-back range: save pre-deletion record (with spent info), then delete.
  - No coins to delete: no-op for this phase.
