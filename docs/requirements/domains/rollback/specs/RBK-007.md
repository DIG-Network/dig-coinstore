# RBK-007: Rollback Atomicity

> **Authoritative requirement:** [RBK-007](../NORMATIVE.md#RBK-007)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

Rollback must be an atomic operation. All steps -- coin deletion (RBK-002), coin un-spending (RBK-003), FF recomputation (RBK-004), hint cleanup (HNT-005), Merkle tree rebuild (RBK-006), and chain tip update -- must either all succeed or all fail. A partial rollback would leave the coinstate in an inconsistent state where some coins are deleted but others are not un-spent, or the Merkle tree root does not match the actual coin state.

This mirrors the atomicity requirement for `apply_block()` (BLK-001) but applied to the reverse operation. The implementation should use a single `WriteBatch` or write transaction that encompasses all rollback mutations.

## Specification

### Atomic Transaction

```rust
pub fn rollback_to_block(&mut self, target_height: i64) -> Result<RollbackResult, CoinStoreError> {
    // Phase 1: Read-only -- identify coins to delete, coins to un-spend
    let coins_to_delete = self.query_coins_confirmed_after(target_height)?;
    let coins_to_unspend = self.query_coins_spent_after(target_height)?;

    // Phase 2: Prepare WriteBatch with all mutations
    let mut write_batch = WriteBatch::new();

    // RBK-002: Delete coins
    for coin in &coins_to_delete {
        write_batch.delete_coin(coin)?;
    }

    // HNT-005: Delete hints for deleted coins
    self.hint_store.remove_hints_to_batch(&mut write_batch, &coin_ids_to_delete)?;

    // RBK-003: Un-spend coins
    for coin in &coins_to_unspend {
        write_batch.unspend_coin(coin)?;
    }

    // RBK-004: Recompute FF eligibility
    // (included in un-spend writes)

    // RBK-006: Merkle tree rebuild
    let dirty_nodes = self.rebuild_merkle(&coins_to_delete, &coins_to_unspend)?;
    for (key, hash) in &dirty_nodes {
        write_batch.put_merkle_node(key, hash);
    }

    // Update chain tip
    write_batch.put_chain_tip(new_tip)?;

    // Atomic commit -- ALL or NOTHING
    self.storage.commit(write_batch)?;

    // Only after successful commit: update in-memory state
    self.apply_rollback_to_memory(&coins_to_delete, &coins_to_unspend, new_tip);

    Ok(rollback_result)
}
```

### Failure Modes

| Failure Point | Expected Behavior |
|--------------|-------------------|
| Storage error during coin deletion query | Return error, no mutations |
| Storage error during un-spend query | Return error, no mutations |
| Error building WriteBatch | Return error, no mutations |
| Storage commit failure | Return error, no mutations committed, no in-memory changes |
| Post-commit in-memory update | Critical error (should not fail; if it does, restart required) |

### In-Memory State

After a successful commit, in-memory state (unspent set, LRU cache, counters, chain tip) MUST be updated to match the durable state. If the commit fails, in-memory state MUST remain at the pre-rollback state.

---

## Acceptance Criteria

- [ ] All rollback mutations are part of a single atomic transaction.
- [ ] If coin deletion fails, no coins are un-spent and no Merkle nodes are modified.
- [ ] If un-spending fails, coin deletions are not committed.
- [ ] If Merkle rebuild fails, coin changes are not committed.
- [ ] If the storage commit fails, no durable or in-memory state changes occur.
- [ ] On success, all durable and in-memory state is consistent.
- [ ] `self.height()` returns the pre-rollback height on failure.
- [ ] `self.height()` returns the target height on success.
- [ ] Partial rollback state is never observable by callers.

## Implementation Notes

- **Primary file:** `src/rollback.rs`
- **Dependencies:** RBK-002 (deletion), RBK-003 (un-spending), RBK-004 (FF recomputation), RBK-006 (Merkle rebuild), HNT-005 (hint cleanup), STO-001 (storage commit)
- **Pattern:** Gather all mutations into a `WriteBatch`, commit once. Prepare in-memory changes but only apply after successful commit.
- **Edge cases:**
  - Empty rollback (no coins affected): commit is a no-op, but chain tip still updated.
  - I/O error mid-commit: storage backend must guarantee atomicity (e.g., redb write transactions).
  - Crash after commit but before in-memory update: on restart, in-memory state rebuilt from durable storage.
