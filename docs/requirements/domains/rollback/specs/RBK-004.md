# RBK-004: Fast-Forward Eligibility Recomputation

> **Authoritative requirement:** [RBK-004](../NORMATIVE.md#RBK-004)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

When a coin is un-spent during rollback, its fast-forward (FF) eligibility flag must be recomputed. FF eligibility is a singleton optimization that allows certain coins to be spent without full CLVM re-execution. The eligibility depends on the coin's relationship with its parent: it must not be a coinbase reward, its parent must exist in the store with the same puzzle hash and amount, and the parent must currently be in a spent state.

This corresponds to the FF eligibility check in Chia L1's rollback logic, which examines parent coin properties to determine if the singleton lineage pattern holds.

---

## Specification

### FF Eligibility Rules

For each coin being un-spent during rollback, compute `ff_eligible` as follows:

```
ff_eligible = (coinbase == false)
           && parent_coin exists in store
           && parent_coin.puzzle_hash == coin.puzzle_hash
           && parent_coin.amount == coin.amount
           && parent_coin.spent_height.is_some()
```

If any condition fails, `ff_eligible = false`.

### Behavior

1. For each coin un-spent in RBK-003:
   a. If `coin.coinbase == true`, set `ff_eligible = false`.
   b. Look up the parent coin by `coin.parent_coin_info`.
   c. If parent not found, set `ff_eligible = false`.
   d. If parent's `puzzle_hash != coin.puzzle_hash` or parent's `amount != coin.amount`, set `ff_eligible = false`.
   e. If parent is unspent (`spent_height == None`), set `ff_eligible = false`.
   f. Otherwise, set `ff_eligible = true`.
2. Persist the updated `ff_eligible` flag in the coin record.

---

## Acceptance Criteria

- [ ] Coinbase coins always have `ff_eligible = false` after rollback.
- [ ] Non-coinbase coins with a matching spent parent have `ff_eligible = true`.
- [ ] Non-coinbase coins with no parent in store have `ff_eligible = false`.
- [ ] Non-coinbase coins with parent having different puzzle_hash or amount have `ff_eligible = false`.
- [ ] Non-coinbase coins with an unspent parent have `ff_eligible = false`.
- [ ] The `ff_eligible` flag is persisted to storage.

---

## Implementation Notes

- **Primary file:** `src/rollback.rs`
- **Dependencies:** RBK-003 (triggers recomputation), STO-001 (parent lookup)
- **Edge cases:**
  - Parent coin may itself have been un-spent in the same rollback, so the order of un-spending matters. The parent's post-rollback state should be used.
  - Orphaned coins (parent deleted in same rollback) will have `ff_eligible = false`.
