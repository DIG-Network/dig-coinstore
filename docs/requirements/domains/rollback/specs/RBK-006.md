# RBK-006: Merkle Tree Batch Rebuild During Rollback

> **Authoritative requirement:** [RBK-006](../NORMATIVE.md#RBK-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

---

## Summary

During rollback, the Merkle tree must be updated to reflect the reverted coin state. Coins that are deleted (RBK-002) must have their leaves removed from the tree. Coins that are un-spent (RBK-003) must have their leaves re-hashed to reflect the new `spent_height = None` state.

These Merkle updates MUST use batch operations to produce a single root recomputation, rather than updating the tree one leaf at a time. This is critical for performance: a rollback of many blocks may affect thousands of coins, and individual root recomputations would be prohibitively expensive.

The resulting dirty Merkle nodes MUST be included in the same write transaction as the coin deletions and un-spending updates (see RBK-007 for atomicity).

---

## Specification

### Batch Update Logic

```rust
let mut merkle_removals: Vec<LeafKey> = Vec::new();
let mut merkle_updates: Vec<(LeafKey, LeafHash)> = Vec::new();

// Deleted coins: remove from Merkle tree
for coin_id in &coins_to_delete {
    merkle_removals.push(*coin_id);
}

// Un-spent coins: re-hash with spent_height = None
for record in &coins_to_unspend {
    let mut updated_record = record.clone();
    updated_record.spent_height = None;
    merkle_updates.push((record.coin_id, updated_record.leaf_hash()));
}

// Single batch operation
let (new_root, dirty_nodes) = self.merkle_tree.batch_update(
    &merkle_removals,
    &merkle_updates,
)?;
```

### WriteBatch Integration

Dirty Merkle nodes from the batch update MUST be part of the same write transaction as:
- Coin record deletions (RBK-002)
- Coin record un-spending updates (RBK-003)
- Hint deletions (HNT-005)
- Chain tip metadata update

### Ordering

The Merkle tree update runs after:
1. RBK-002 (coin deletion) -- identifies leaves to remove
2. RBK-003 (coin un-spending) -- identifies leaves to re-hash
3. RBK-004 (FF recomputation) -- may affect leaf hash if ff_eligible is part of the hash

And before:
4. Chain tip update
5. Transaction commit

## Acceptance Criteria

- [ ] Deleted coins have their Merkle leaves removed.
- [ ] Un-spent coins have their Merkle leaves re-hashed with `spent_height = None`.
- [ ] A single root recomputation occurs for the entire rollback.
- [ ] Dirty Merkle nodes are written in the same transaction as coin changes.
- [ ] After rollback, `self.merkle_tree.root()` reflects the reverted state.
- [ ] The Merkle root after rollback to height N equals the root that was computed when height N was the tip.
- [ ] A rollback with no affected coins produces no Merkle updates.

## Implementation Notes

- **Primary file:** `src/rollback.rs`, `src/merkle.rs`
- **Dependencies:** MRK-001 (Merkle tree structure), RBK-002 (deletion list), RBK-003 (un-spending list), RBK-007 (atomicity)
- **Edge cases:**
  - Rollback with only deletions (no un-spending): only removals from tree.
  - Rollback with only un-spending (no deletions): only re-hashes.
  - Full reset (negative target_height): all leaves removed, tree returns to empty root.
  - Large rollback (many blocks): batch operation handles efficiently.
