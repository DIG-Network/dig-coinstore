# BLK-011: Hint Validation in Phase 1

> **Authoritative requirement:** [BLK-011](../NORMATIVE.md#BLK-011)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) -- Section 5, Section 8

---

## Summary

`apply_block()` must validate all hints in `block.hints` during Phase 1 (read-only validation), before any Phase 2 mutations begin. This ensures that a block containing a malformed hint is rejected cleanly without leaving partial state changes.

Empty hints (0 bytes) are semantically meaningless and MUST be silently skipped -- they carry no puzzle-hash information for subscription matching. Hints exceeding `MAX_HINT_LENGTH` (32 bytes) are malformed and MUST cause the entire block to be rejected. This fail-fast approach prevents storing garbage data and aligns with HNT-001 validation rules.

By performing hint validation in Phase 1, the implementation guarantees that Phase 2 hint storage (BLK-012) only operates on pre-validated data, simplifying error handling in the mutation path.

---

## Specification

### Validation Logic

```rust
for (coin_id, hint) in &block.hints {
    if hint.is_empty() {
        continue; // silently skip empty hints
    }
    if hint.len() > MAX_HINT_LENGTH {
        return Err(CoinStoreError::HintTooLong {
            coin_id: *coin_id,
            length: hint.len(),
            max: MAX_HINT_LENGTH,
        });
    }
}
```

### Constants

| Constant | Value | Source |
|----------|-------|--------|
| `MAX_HINT_LENGTH` | `32` | HNT-001, SPEC Section 8 |

### Error Type

```rust
CoinStoreError::HintTooLong {
    coin_id: Bytes32,
    length: usize,
    max: usize,
}
```

### Ordering

This check runs in Phase 1, after BLK-006 (addition validation) and before any Phase 2 mutations. The exact position within Phase 1 is after all coin-level checks are complete.

### Pipeline Position

```
Phase 1 (Validation):
  BLK-002 -> BLK-003 -> BLK-004 -> BLK-005 -> BLK-006 -> BLK-011 (hint validation)
Phase 2 (Mutation):
  BLK-007 -> BLK-008 -> BLK-012 (hint storage) -> BLK-013 -> BLK-009 -> BLK-014
```

---

## Acceptance Criteria

- [ ] All hints in `block.hints` are validated during Phase 1.
- [ ] Empty hints (0 bytes) are silently skipped without error.
- [ ] Hints with `len() > 32` cause the entire block to be rejected with `CoinStoreError::HintTooLong`.
- [ ] Hints with `1 <= len() <= 32` pass validation.
- [ ] No mutations occur before hint validation completes (Phase 1 guarantee).
- [ ] A block with mixed valid and invalid hints is fully rejected.
- [ ] The error includes the offending coin_id and hint length for diagnostics.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`)
- **Dependencies:** HNT-001 (defines validation rules), BLK-001 (pipeline orchestration), BLK-012 (Phase 2 storage depends on this validation)
- **Edge cases:**
  - Block with empty `hints` list: no validation needed, passes trivially.
  - Block with only empty hints: all skipped, passes.
  - Block with one oversized hint among many valid hints: entire block rejected.
  - Multiple hints per coin_id: each validated independently.
  - Hint of exactly 32 bytes: valid, not rejected.
