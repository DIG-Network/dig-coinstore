# BLK-002: Height Continuity Check

> **Authoritative requirement:** [BLK-002](../NORMATIVE.md#BLK-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The block height must be exactly `current_height + 1`. This ensures blocks are applied in strict sequential order with no gaps, no regressions, and no skips. Any deviation is rejected immediately as the first validation check in Phase 1.

This is fundamental to the chain model: the coinstate maintains a monotonically increasing height counter, and each block extends the chain by exactly one. A gap would mean missing state transitions; a regression would mean re-applying a historical block; a duplicate would mean double-applying the same height.

Chia L1 performs this check implicitly through the blockchain's consensus layer before calling `CoinStore.new_block()`. dig-coinstore makes it explicit within `apply_block()` as a defense-in-depth measure.

---

## Specification

### Validation Logic

```rust
if block.height != self.height() + 1 {
    return Err(CoinStoreError::HeightMismatch {
        expected: self.height() + 1,
        got: block.height,
    });
}
```

### Error Type

```rust
CoinStoreError::HeightMismatch {
    expected: u64,  // self.height() + 1
    got: u64,       // block.height
}
```

### Ordering

This MUST be the first validation check in Phase 1. If the height is wrong, there is no point in validating parent hash, removals, or additions.

### Genesis Case

When the coinstate is freshly initialized (no blocks applied), `self.height()` returns a sentinel value (e.g., `u64::MAX` or a separate `is_empty()` check). The genesis block at height 0 must be handled correctly.

---

## Acceptance Criteria

- [ ] `block.height == self.height() + 1` is enforced.
- [ ] Height mismatch returns `CoinStoreError::HeightMismatch { expected, got }`.
- [ ] Sequential blocks (0, 1, 2, ...) are accepted.
- [ ] A gap (e.g., height 0 then height 2) is rejected.
- [ ] A regression (e.g., height 5 then height 3) is rejected.
- [ ] A duplicate (e.g., height 5 then height 5) is rejected.
- [ ] Genesis block (height 0) is accepted on a fresh coinstate.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`)
- **Dependencies:** BLK-001 (entry point that calls this check)
- **Edge cases:**
  - Fresh coinstate with no blocks: `self.height()` must be defined such that `self.height() + 1 == 0` for genesis.
  - `u64::MAX` height: Overflow of `self.height() + 1` is unreachable in practice (would require applying 2^64 blocks).

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| `test_height_sequential_accepted` | Integration | Apply blocks 0, 1, 2 in order | All succeed |
| `test_height_gap_rejected` | Integration | Apply block 0, then block 2 | `Err(HeightMismatch { expected: 1, got: 2 })` |

| `test_height_non_zero_on_fresh` | Integration | Apply block 1 on fresh coinstate | `Err(HeightMismatch { expected: 0, got: 1 })` |
| `test_height_overflow` | Unit | Apply block at height u64::MAX, then attempt height u64::MAX + 1 (overflow) | Handled gracefully: either error or unreachable in practice |
