# BLK-012: Hint Storage in Phase 2

> **Authoritative requirement:** [BLK-012](../NORMATIVE.md#BLK-012)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) -- Section 5, Section 8

---

## Summary

After all Phase 1 validation passes (including BLK-011 hint validation), `apply_block()` must store all validated hints from `block.hints` during Phase 2. Hint storage operations must be included in the same `WriteBatch` as coin record writes (BLK-007, BLK-008) to ensure atomicity. If the block application fails and is rolled back, hint writes are also rolled back.

Duplicate `(coin_id, hint)` pairs must be silently ignored, matching HNT-002 idempotent insertion semantics. This is important because the same block may be re-processed after a rollback.

---

## Specification

### Storage Logic

```rust
// Phase 2: After coin insertion (BLK-007) and spend marking (BLK-008)
for (coin_id, hint) in &validated_hints {
    // hint was already validated in Phase 1 (BLK-011)
    // Empty hints were filtered out during validation
    self.hint_store.add_hint_to_batch(&mut write_batch, coin_id, hint)?;
}
```

### WriteBatch Integration

All hint writes MUST be added to the same `WriteBatch` that contains:
- Coin record insertions (BLK-007)
- Spend marking updates (BLK-008)
- Merkle tree dirty nodes (BLK-013)
- Chain tip metadata (BLK-014)

This ensures that a failed commit rolls back all state changes atomically, including hints.

### Idempotent Semantics

If a `(coin_id, hint)` pair already exists in the hint store (e.g., from a prior application of the same block before a rollback), the insert MUST be silently ignored. No error, no duplicate entries.

### Ordering within Phase 2

Hint storage runs after coin insertion (BLK-007) and spend marking (BLK-008), but before state root verification (BLK-009). This ensures hints reference coins that actually exist in the store.

---

## Acceptance Criteria

- [ ] All validated hints from `block.hints` are written to the hint store during Phase 2.
- [ ] Hint writes are part of the same `WriteBatch` as coin record writes.
- [ ] Duplicate `(coin_id, hint)` pairs are silently ignored.
- [ ] If the block application fails after hint storage, hints are rolled back with everything else.
- [ ] Empty hints (filtered in BLK-011) are not stored.
- [ ] Both forward and reverse indices are updated (see HNT-003).
- [ ] Hints are queryable via `get_hints_for_coin_ids()` after successful block application.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`), `src/hints.rs`
- **Dependencies:** BLK-011 (validation), HNT-002 (idempotent insertion), HNT-003 (bidirectional indices), BLK-007/008 (same WriteBatch)
- **Edge cases:**
  - Block with no hints: no hint writes, WriteBatch contains only coin operations.
  - Block with all empty hints (filtered out in BLK-011): no hint writes.
  - Re-application after rollback: all hints re-inserted idempotently.
  - Large number of hints: batched into the WriteBatch efficiently.
