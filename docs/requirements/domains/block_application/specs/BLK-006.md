# BLK-006: Addition Validation

> **Authoritative requirement:** [BLK-006](../NORMATIVE.md#BLK-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.6

---

## Summary

No coin in `block.additions` (including coinbase coins) should already exist in the coinstate. While coin ID collisions are cryptographically unlikely (would require a SHA256 collision on the `parent_coin_id || puzzle_hash || amount` triple), this check catches logic errors such as double-processing a block or corrupted `BlockData`.

This is a Phase 1 read-only check. If any addition already exists, the entire block is rejected before any mutations occur.

---

## Specification

### Validation Logic

```rust
for addition in &block.additions {
    if self.get_coin_record(&addition.coin_id)?.is_some() {
        return Err(CoinStoreError::CoinAlreadyExists(addition.coin_id));
    }
}

for coinbase in &block.coinbase_coins {
    if self.get_coin_record(&coinbase.coin_id)?.is_some() {
        return Err(CoinStoreError::CoinAlreadyExists(coinbase.coin_id));
    }
}
```

### Error Type

```rust
CoinStoreError::CoinAlreadyExists(Bytes32)  // coin_id already in coinstate
```

### Ordering

This check runs in Phase 1 after BLK-005 (removal validation). It covers both transaction additions and coinbase coins.

### Scope

Both `block.additions` and `block.coinbase_coins` are validated. A coin ID collision in either list triggers rejection.

---

## Acceptance Criteria

- [ ] Every coin_id in additions is checked for non-existence.
- [ ] Every coin_id in coinbase_coins is checked for non-existence.
- [ ] An existing coin_id returns `CoinStoreError::CoinAlreadyExists(coin_id)`.
- [ ] New coin_ids pass this check.
- [ ] Empty additions list passes trivially.
- [ ] A block with one duplicate addition rejects the entire block.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`)
- **Dependencies:** BLK-001 (entry point), BLK-005 (runs before this check)
- **Edge cases:**
  - Coin ID collision between additions and coinbase_coins in the same block: This is a caller error. The validation checks both lists against existing state; intra-block collision detection is an optional additional check.
  - Coin previously added and then removed (spent) in a prior block: The coin record still exists (with spent_height set), so `get_coin_record` returns `Some`. This correctly rejects a re-addition of the same coin_id.
