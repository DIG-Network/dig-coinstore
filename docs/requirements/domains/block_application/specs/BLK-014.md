# BLK-014: Chain Tip and Atomic Commit

> **Authoritative requirement:** [BLK-014](../NORMATIVE.md#BLK-014)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) -- Section 5

---

## Summary

The final step of Phase 2 in `apply_block()` is to atomically update the chain tip metadata and commit all mutations as a single unit. The chain tip includes the current height, tip hash, and timestamp. These fields MUST be written to the same `WriteBatch` as all other Phase 2 mutations (coin records, hints, Merkle nodes).

After the `WriteBatch` is successfully committed to durable storage, the in-memory state (unspent set, LRU cache, materialized counters, chain tip fields) MUST be swapped atomically. This two-phase approach ensures that:
1. If the disk write fails, no in-memory state is modified.
2. If the disk write succeeds, in-memory state is guaranteed to reflect the committed data.

## Specification

### Chain Tip Metadata

| Field | Value | Description |
|-------|-------|-------------|
| `height` | `block.height` | The new chain height after applying this block |
| `tip_hash` | `block.block_hash` | The hash of the block just applied |
| `timestamp` | `block.timestamp` | The timestamp of the block just applied |

### WriteBatch Contents

The final `WriteBatch` committed during Phase 2 contains:

1. Coin record insertions (BLK-007)
2. Spend marking updates (BLK-008)
3. Hint writes (BLK-012)
4. Dirty Merkle nodes (BLK-013)
5. **Chain tip metadata (this requirement)**

### Commit and Swap Logic

```rust
// Write chain tip to the batch
write_batch.put_chain_tip(ChainTip {
    height: block.height,
    tip_hash: block.block_hash,
    timestamp: block.timestamp,
});

// Atomically commit all Phase 2 mutations to storage
self.storage.commit(write_batch)?;

// Only after successful commit: swap in-memory state
self.unspent_set.apply_pending(pending_unspent_changes);
self.lru_cache.apply_pending(pending_cache_changes);
self.counters.apply_pending(pending_counter_changes);
self.chain_tip = ChainTip {
    height: block.height,
    tip_hash: block.block_hash,
    timestamp: block.timestamp,
};
```

### Atomicity of In-Memory Swap

The in-memory swap MUST be all-or-nothing. If the implementation uses multiple in-memory structures, they must all be updated together. A common pattern is to prepare all pending changes before the commit, then apply them in sequence after the commit succeeds. Since this runs under the `&mut self` exclusive lock, no concurrent readers can observe a partial in-memory state.

### Failure Semantics

- **Commit failure:** `storage.commit()` returns an error. No in-memory state is modified. The `CoinStore` remains at the previous height. The caller may retry with a new block.
- **Post-commit crash:** On restart, the durable state reflects the committed block. The in-memory state is rebuilt from durable storage during initialization.

## Acceptance Criteria

- [ ] Chain tip metadata (height, tip_hash, timestamp) is written to the `WriteBatch`.
- [ ] The `WriteBatch` is committed atomically (all or nothing).
- [ ] On successful commit, `self.height()` returns the new height.
- [ ] On successful commit, `self.tip_hash()` returns the new block hash.
- [ ] On successful commit, in-memory unspent set reflects all additions and removals.
- [ ] On successful commit, materialized counters are updated.
- [ ] On commit failure, no in-memory state is modified.
- [ ] On commit failure, `self.height()` returns the pre-apply height.
- [ ] The LRU cache is updated after successful commit.

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`)
- **Dependencies:** BLK-007 (coin insertions in batch), BLK-008 (spend marks in batch), BLK-012 (hints in batch), BLK-013 (Merkle nodes in batch), STO-001 (storage commit)
- **Edge cases:**
  - Storage full / I/O error during commit: must return error, no state changes.
  - Genesis block commit: first chain tip write (height 0).
  - Very large WriteBatch (large block): storage layer must handle efficiently.
  - Concurrent readers during in-memory swap: not possible due to `&mut self` exclusive borrow.
