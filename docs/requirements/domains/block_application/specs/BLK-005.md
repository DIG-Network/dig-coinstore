# BLK-005: Removal Validation

> **Authoritative requirement:** [BLK-005](../NORMATIVE.md#BLK-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.5

## Summary

Every coin ID in `block.removals` must exist in the coinstate and be currently unspent. This validation occurs entirely in Phase 1 (read-only), **before** any mutations take place. This ensures atomicity: if any removal is invalid, the block is rejected without modifying state.

Chia performs the equivalent check in `_set_spent()` ([`coin_store.py:627-648`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/coin_store.py#L627)), where `rows_updated != len(coin_names)` raises `ValueError`. dig-coinstore adds an explicit pre-mutation validation pass for clarity and to distinguish between "coin not found" and "coin already spent" error cases.

---

## Specification

### Validation Logic

```rust
for coin_id in &block.removals {
    let record = self.get_coin_record(coin_id)?
        .ok_or(CoinStoreError::CoinNotFound(*coin_id))?;
    if record.is_spent() {
        return Err(CoinStoreError::DoubleSpend(*coin_id));
    }
}
```

### Error Types

```rust
CoinStoreError::CoinNotFound(Bytes32)   // coin_id does not exist in coinstate
CoinStoreError::DoubleSpend(Bytes32)    // coin_id exists but is already spent
```

### Ordering

This check runs in Phase 1 after BLK-004 (reward coin assertion). It validates all removals before any Phase 2 mutations begin.

### Pre-Mutation Guarantee

The entire removal list is validated before any coin is inserted (BLK-007) or spent (BLK-008). This means:
- A block cannot partially apply (some removals valid, some not).
- Phase 2 mutations can assume all removals are valid.
- There is no TOCTOU race within a single `apply_block()` call.

### Duplicate Removal Detection

If the same coin_id appears twice in `block.removals`, the second occurrence would pass this check (it is still unspent in Phase 1). The duplicate is caught by BLK-008's strict count assertion or by deduplication at the caller level.

---

## Acceptance Criteria

- [ ] Every coin_id in removals is checked for existence.
- [ ] Missing coins return `CoinStoreError::CoinNotFound(coin_id)`.
- [ ] Already-spent coins return `CoinStoreError::DoubleSpend(coin_id)`.
- [ ] Validation occurs entirely before mutations (Phase 1).
- [ ] A block with valid removals passes this check.
- [ ] A block with one invalid removal rejects the entire block.
- [ ] Empty removals list passes trivially.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`)
- **Dependencies:** BLK-001 (entry point), BLK-008 (Phase 2 spend marking provides defense-in-depth)
- **Edge cases:**
  - Empty removals: Valid. Skip the loop.
  - Coin created in the same block's additions: Not yet inserted (Phase 1 is read-only), so would return CoinNotFound. Intra-block spend-of-creation is not supported.
  - Duplicate coin_id in removals: Passes Phase 1 (coin is unspent for both checks). Caught by BLK-008 count assertion.
