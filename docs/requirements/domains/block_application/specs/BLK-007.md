## Summary

Transaction additions and coinbase coins are inserted as new `CoinRecord` entries during Phase 2 (mutation). Each record is created with the block's height, no spent marker, and the block's timestamp. A key feature is **fast-forward (FF) eligible tracking at creation time**: coins that have the same puzzle_hash and amount as their parent (singleton pattern) are flagged immediately, avoiding a costly post-hoc migration scan.

This is adopted from Chia L1's `CoinStore.new_block()` ([`coin_store.py:128-129`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/coin_store.py#L128)), where `spent_index = -1` marks a coin as FF-eligible (vs `0` for normal unspent). dig-coinstore uses an explicit `ff_eligible: bool` flag on the `CoinRecord` for clarity.

The `same_as_parent` flag is computed by the caller (who has access to the parent coin during CLVM execution) and passed in `CoinAddition`. It means: this coin has the same `puzzle_hash` and `amount` as its parent coin and is not a coinbase reward. This pattern is characteristic of singleton spends.

## Specification

### Coin Record Fields

| Field | Value | Source |
|-------|-------|--------|
| `coin_id` | SHA256(parent_coin_id, puzzle_hash, amount) | Computed by caller |
| `confirmed_height` | `block.height` | Block being applied |
| `spent_height` | `None` | Unspent at creation |
| `timestamp` | `block.timestamp` | Block being applied |
| `coinbase` | `false` for additions, `true` for coinbase_coins | Type of coin |
| `ff_eligible` | `true` if `same_as_parent && !coinbase` | See FF-eligible rules |

### FF-Eligible Rules

```rust
// Transaction additions
for addition in &block.additions {
    let ff_eligible = addition.same_as_parent;  // caller-determined
    self.insert_coin_record(CoinRecord {
        coin_id: addition.coin_id,
        confirmed_height: block.height,
        spent_height: None,
        timestamp: block.timestamp,
        coinbase: false,
        ff_eligible,
        // ... coin data fields
    })?;
}

// Coinbase coins: NEVER FF-eligible
for coinbase in &block.coinbase_coins {
    self.insert_coin_record(CoinRecord {
        coin_id: coinbase.coin_id,
        confirmed_height: block.height,
        spent_height: None,
        timestamp: block.timestamp,
        coinbase: true,
        ff_eligible: false,  // always false for coinbase
        // ... coin data fields
    })?;
}
```

### Batching

Chia inserts all records in a single `executemany` ([`coin_store.py:161`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/coin_store.py#L161)). dig-coinstore should similarly batch the write for performance.

## Acceptance Criteria

- [ ] Transaction additions are stored with `confirmed_height = block.height`.
- [ ] Transaction additions are stored with `spent_height = None`.
- [ ] Transaction additions are stored with `timestamp = block.timestamp`.
- [ ] Transaction additions are stored with `coinbase = false`.
- [ ] When `same_as_parent = true`, `ff_eligible = true` is set.
- [ ] When `same_as_parent = false`, `ff_eligible = false` is set.
- [ ] Coinbase coins are stored with `coinbase = true`.
- [ ] Coinbase coins always have `ff_eligible = false`, regardless of any other flags.
- [ ] All coins are retrievable via `get_coin_record()` after insertion.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`)
- **Dependencies:** BLK-006 (validation must pass before insertion), BLK-001 (entry point)
- **Edge cases:**
  - `same_as_parent = true` on a coinbase coin: The coinbase path always sets `ff_eligible = false`, overriding any flag.
  - Empty additions and coinbase lists: Valid. No insertions needed.
  - Large addition sets: Batch insertion should be used for performance.
