# BLK-008: Spend Marking with Strict Count Assertion

## Summary

All removal coins are marked as spent at the current block height during Phase 2 (mutation). The spend operation uses a two-layer safety pattern adopted from Chia L1's `_set_spent()` ([`coin_store.py:627-648`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/coin_store.py#L627)):

1. **WHERE guard**: Only updates records where `spent_height` is `None` (unspent). Already-spent coins are silently skipped by the update.
2. **Post-mutation count assertion**: After all spend updates, the number of rows actually modified must equal the number of removals. If they differ, something went wrong.

This defense-in-depth approach catches edge cases that the Phase 1 validation (BLK-005) might miss due to TOCTOU races in concurrent environments.

In Chia L1: `WHERE spent_index <= 0` ([`coin_store.py:640-641`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/coin_store.py#L640)) and `if rows_updated != len(coin_names): raise ValueError(...)` ([`coin_store.py:645-648`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/coin_store.py#L645)).

## Specification

### Spend Marking Logic

```rust
let rows_updated = self.storage.mark_coins_spent(&block.removals, block.height)?;
if rows_updated != block.removals.len() {
    return Err(CoinStoreError::SpendCountMismatch {
        expected: block.removals.len(),
        actual: rows_updated,
    });
}
```

### WHERE Guard (Storage Layer)

The storage layer's `mark_coins_spent()` implementation must use a conditional update that only modifies records where `spent_height` is `None`:

```sql
-- Conceptual SQL equivalent
UPDATE coin_records
SET spent_height = ?
WHERE coin_id IN (?, ?, ...)
  AND spent_height IS NULL
```

The function returns the number of rows actually modified.

### Error Type

```rust
CoinStoreError::SpendCountMismatch {
    expected: usize,  // block.removals.len()
    actual: usize,    // rows_updated from storage
}
```

### Batching

For large removal sets, processing occurs in chunks of `DEFAULT_LOOKUP_BATCH_SIZE` to bound intermediate memory allocation. This is adopted from Chia's batching at [`coin_store.py:635`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/coin_store.py#L635).

### Ordering

Spend marking runs in Phase 2 after coin insertion (BLK-007). This ordering is important: if an addition and a removal in the same block refer to different coins, both operations succeed. Intra-block spend-of-creation is not supported (the removal must reference a coin from a prior block).

## Acceptance Criteria

- [ ] All removal coins are marked with `spent_height = block.height`.
- [ ] The WHERE guard only updates records where `spent_height` is `None`.
- [ ] `rows_updated` is returned from the storage layer.
- [ ] `rows_updated != removals.len()` returns `CoinStoreError::SpendCountMismatch`.
- [ ] `rows_updated == removals.len()` proceeds without error.
- [ ] Empty removals list: `rows_updated == 0 == removals.len()`, passes.
- [ ] After spending, `get_coin_record()` returns a record with `spent_height = Some(block.height)`.

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/storage.rs` for the storage layer)
- **Dependencies:** BLK-005 (Phase 1 validation), BLK-007 (coin insertion runs before this)
- **Edge cases:**
  - Duplicate coin_id in removals: The first occurrence sets `spent_height`. The second occurrence's WHERE guard finds `spent_height IS NOT NULL`, so it silently skips. `rows_updated` will be less than `removals.len()`, triggering SpendCountMismatch. This correctly rejects duplicate removals.
  - Concurrent block application: If two threads try to apply different blocks spending the same coin, the WHERE guard ensures only one succeeds. The count assertion catches the other.
  - Empty removals: Valid. `mark_coins_spent` returns 0, which equals `removals.len()` (0).
