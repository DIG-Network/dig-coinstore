# BLK-003: Parent Hash Verification

> **Authoritative requirement:** [BLK-003](../NORMATIVE.md#BLK-003)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.3

---

## Summary

The block's `parent_hash` must match the hash of the current chain tip block. This ensures the block forms a valid chain extension and prevents applying blocks from a different fork or out-of-order chain.

For the genesis block (height 0), there is no parent, so the parent hash must be the zero hash (`[0u8; 32]`). This is a well-known convention in blockchain systems.

Together with BLK-002 (height continuity), the parent hash check provides a two-layer chain integrity guarantee: height ensures sequential ordering, and parent hash ensures cryptographic linkage.

---

## Specification

### Validation Logic

```rust
if block.parent_hash != self.tip_hash() {
    return Err(CoinStoreError::ParentHashMismatch {
        expected: self.tip_hash(),
        got: block.parent_hash,
    });
}
```

### Error Type

```rust
CoinStoreError::ParentHashMismatch {
    expected: Bytes32,  // self.tip_hash()
    got: Bytes32,       // block.parent_hash
}
```

### Genesis Case

For the genesis block (height 0), `self.tip_hash()` returns the zero hash (`Bytes32::default()` or equivalent `[0u8; 32]`). The genesis block's `parent_hash` must also be the zero hash.

### Ordering

This check runs immediately after BLK-002 (height continuity) in Phase 1. If height is wrong, parent hash is not checked.

---

## Acceptance Criteria

- [ ] `block.parent_hash == self.tip_hash()` is enforced.
- [ ] Parent hash mismatch returns `CoinStoreError::ParentHashMismatch { expected, got }`.
- [ ] A block with the correct parent hash is accepted.
- [ ] A block with an incorrect parent hash is rejected.
- [ ] Genesis block with zero hash parent is accepted.
- [ ] Genesis block with non-zero parent hash is rejected.
- [ ] After applying block N, `self.tip_hash()` returns block N's `block_hash`, which must be block N+1's `parent_hash`.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`)
- **Dependencies:** BLK-001 (entry point), BLK-002 (runs before this check)
- **Edge cases:**
  - Fresh coinstate: `tip_hash()` must return zero hash for genesis compatibility.
  - Fork detection: If a block has the right height but wrong parent hash, it belongs to a different fork. The caller must handle reorgs before retrying.
