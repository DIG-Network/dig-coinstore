# BLK-013: Merkle Tree Batch Update in Phase 2

> **Authoritative requirement:** [BLK-013](../NORMATIVE.md#BLK-013)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) -- Section 5, MRK-001, MRK-003

---

## Summary

During Phase 2 of `apply_block()`, the Merkle tree must be updated to reflect all coin state changes from the current block. Rather than updating the tree one leaf at a time (which would require O(n * log(n)) root recomputations), the implementation MUST use `SparseMerkleTree::batch_insert()` to process all new and modified coin record hashes in a single batch operation, producing a single root recomputation.

Dirty Merkle nodes (internal nodes whose hashes changed due to leaf updates) MUST be included in the same `WriteBatch` as coin record writes, hint writes, and chain tip metadata. This ensures that the Merkle tree is always consistent with the coin state.

---

## Specification

### Batch Update Logic

```rust
// Collect all leaf updates for this block
let mut merkle_updates: Vec<(LeafKey, LeafHash)> = Vec::new();

// New coins (additions + coinbase)
for record in &new_coin_records {
    merkle_updates.push((record.coin_id, record.leaf_hash()));
}

// Modified coins (removals: spent_height changed)
for record in &spent_coin_records {
    merkle_updates.push((record.coin_id, record.leaf_hash()));
}

// Single batch operation: one root recomputation
let (new_root, dirty_nodes) = self.merkle_tree.batch_insert(&merkle_updates)?;

// Add dirty nodes to the WriteBatch
for (node_key, node_hash) in &dirty_nodes {
    write_batch.put_merkle_node(node_key, node_hash);
}
```

### WriteBatch Integration

Dirty Merkle nodes MUST be written to the same `WriteBatch` as:
- Coin record insertions (BLK-007)
- Spend marking updates (BLK-008)
- Hint writes (BLK-012)
- Chain tip metadata (BLK-014)

### Performance

Using `batch_insert()` amortizes the cost of Merkle tree updates:
- **Without batching:** O(n * depth) hash computations, n root recomputations.
- **With batching:** O(n * depth) hash computations, 1 root recomputation.

For sparse Merkle trees with depth 256, this is a significant optimization when blocks contain thousands of coin changes.

### Ordering within Phase 2

Merkle tree update runs after:
1. BLK-007 (coin insertion) -- new leaf hashes available
2. BLK-008 (spend marking) -- modified leaf hashes available
3. BLK-012 (hint storage)

And before:
4. BLK-009 (state root verification) -- requires the updated root
5. BLK-014 (chain tip commit)

## Acceptance Criteria

- [ ] `SparseMerkleTree::batch_insert()` is used for Merkle updates during `apply_block()`.
- [ ] All new coin record hashes are included in the batch.
- [ ] All modified (spent) coin record hashes are included in the batch.
- [ ] A single root recomputation occurs per block.
- [ ] Dirty Merkle nodes are written to the same `WriteBatch` as coin records.
- [ ] The computed root is available for BLK-009 state root verification.
- [ ] An empty block (no additions, no removals) produces no Merkle updates.
- [ ] The Merkle tree root after update is deterministic given the same coin state.

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`), `src/merkle.rs`
- **Dependencies:** MRK-001 (Merkle tree structure), MRK-003 (batch operations), BLK-007/008 (provide leaf data)
- **Edge cases:**
  - Block with only additions (no removals): only new leaves inserted.
  - Block with only removals (no additions): only existing leaves updated with new spent_height hashes.
  - Genesis block: first leaves inserted into an empty tree.
  - Large blocks (10k+ coins): batch_insert handles efficiently with single root recomputation.
  - Merkle tree depth: sparse Merkle tree uses 256-bit key space (coin_id as key).
