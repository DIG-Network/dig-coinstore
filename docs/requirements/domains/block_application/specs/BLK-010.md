## Summary

Block application is timed from start to finish. If the elapsed time exceeds a configurable warning threshold, a warning is logged with diagnostic information. This early warning helps operators identify storage bottlenecks before they cause consensus delays.

This is adopted from Chia L1's `CoinStore.new_block()` ([`coin_store.py:164-178`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/coin_store.py#L164)):

```python
time_end = time.time()
if time_end - time_start > BLOCK_APPLY_WARN_SECONDS:
    log.warning(
        f"Height {height}: block application took {time_end - time_start:.2f}s "
        f"({len(additions)} additions, {len(removals)} removals). "
        f"Ensure coinstate database is on fast storage."
    )
else:
    log.debug(
        f"Height {height}: block application took {time_end - time_start:.2f}s "
        f"({len(additions)} additions, {len(removals)} removals)."
    )
```

## Specification

### Timing Logic

```rust
let start = std::time::Instant::now();

// ... entire apply_block pipeline (Phase 1 + Phase 2) ...

let elapsed = start.elapsed();
let additions_count = block.additions.len() + block.coinbase_coins.len();
let removals_count = block.removals.len();

if elapsed.as_secs_f64() > BLOCK_APPLY_WARN_SECONDS {
    warn!(
        "Height {}: block application took {:.2}s ({} additions, {} removals). \
         Ensure coinstate database is on fast storage.",
        block.height,
        elapsed.as_secs_f64(),
        additions_count,
        removals_count,
    );
} else {
    debug!(
        "Height {}: block application took {:.2}s ({} additions, {} removals).",
        block.height,
        elapsed.as_secs_f64(),
        additions_count,
        removals_count,
    );
}
```

### Constants

| Constant | Value | Source |
|----------|-------|--------|
| `BLOCK_APPLY_WARN_SECONDS` | `10.0` (seconds) | SPEC Section 5.14, Chia `coin_store.py:164` |

### Log Format

**Warning (slow block):**
```
WARN: Height {height}: block application took {elapsed:.2}s ({additions} additions, {removals} removals). Ensure coinstate database is on fast storage.
```

**Debug (normal block):**
```
DEBUG: Height {height}: block application took {elapsed:.2}s ({additions} additions, {removals} removals).
```

### Timing Scope

The timer spans the entire `apply_block()` execution: Phase 1 (validation) + Phase 2 (mutation). Phase 3 (this logging step) is the final action.

### Error Path

Performance logging occurs on the **success** path. If `apply_block()` returns an error, logging behavior is implementation-defined (may log the failure at error level, but not the performance warning).

---

## Acceptance Criteria

- [ ] Block application is timed from start to finish using `std::time::Instant`.
- [ ] If elapsed > `BLOCK_APPLY_WARN_SECONDS` (10s), a WARN-level log is emitted.
- [ ] If elapsed <= `BLOCK_APPLY_WARN_SECONDS`, a DEBUG-level log is emitted.
- [ ] The log message includes: block height, elapsed time (2 decimal places), additions count, removals count.
- [ ] The warning message includes the storage guidance text.
- [ ] `BLOCK_APPLY_WARN_SECONDS` is a named constant (not a magic number).

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`)
- **Dependencies:** BLK-001 (entry point wraps the timing)
- **Edge cases:**
  - Very fast blocks (< 1ms): Logged at DEBUG with "0.00s". No special handling needed.
  - Blocks that fail validation: Timer still runs but the performance warning is only meaningful for successful blocks.
  - Clock precision: `std::time::Instant` provides monotonic, high-resolution timing. No concerns about wall-clock adjustments.
