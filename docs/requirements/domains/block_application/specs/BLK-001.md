## Summary

`apply_block()` is the core state transition function of the dig-coinstore crate. It takes a `BlockData` struct (pre-extracted state changes from a validated block) and atomically applies all additions, removals, hints, and Merkle tree updates. This corresponds to Chia L1's `CoinStore.new_block()` ([`coin_store.py:105-178`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/coin_store.py#L105)).

The function orchestrates the full block application pipeline: Phase 1 (validation, read-only), Phase 2 (mutation, atomic), and Phase 3 (observability). If any step in Phase 1 or Phase 2 fails, the entire operation must roll back -- no partial state changes are allowed.

The atomicity guarantee is critical for consensus safety. A partial block application (e.g., additions inserted but removals not marked spent) would leave the coinstate in an inconsistent state that violates conservation invariants.

## Specification

### Function Signature

```rust
impl CoinStore {
    pub fn apply_block(
        &mut self,
        block: BlockData,
    ) -> Result<ApplyBlockResult, CoinStoreError>;
}
```

### Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `block` | `BlockData` | Pre-extracted state changes from a validated block. Contains `height`, `parent_hash`, `block_hash`, `timestamp`, `additions`, `coinbase_coins`, `removals`, `hints`, and `expected_state_root`. Consumed by value. |

### Return Type

- **`Ok(ApplyBlockResult)`** -- Block was successfully applied. Contains the new state root, height, and summary statistics.
- **`Err(CoinStoreError)`** -- Block application failed. No state changes occurred. The error variant indicates the specific failure (HeightMismatch, ParentHashMismatch, CoinNotFound, DoubleSpend, CoinAlreadyExists, SpendCountMismatch, StateRootMismatch).

### Atomicity

The function MUST guarantee all-or-nothing semantics:
- If any validation check in Phase 1 fails, no mutations occur.
- If any mutation in Phase 2 fails, all prior mutations in that phase are rolled back.
- On success, all mutations are committed as a single unit.

### Pipeline Orchestration

`apply_block()` orchestrates the following steps in order:

1. **Phase 1 -- Validation:** BLK-002 (height), BLK-003 (parent hash), BLK-004 (reward coins), BLK-005 (removals), BLK-006 (additions)
2. **Phase 2 -- Mutation:** BLK-007 (coin insertion), BLK-008 (spend marking), BLK-009 (state root verification), chain tip update
3. **Phase 3 -- Observability:** BLK-010 (performance logging)

## Acceptance Criteria

- [ ] The method `apply_block()` exists on the coinstate struct with the exact signature specified above.
- [ ] `BlockData` is consumed by value (owned, not borrowed).
- [ ] Returns `Result<ApplyBlockResult, CoinStoreError>` with correct variants.
- [ ] A valid block with correct additions and removals returns `Ok(ApplyBlockResult)`.
- [ ] A block that fails any validation returns the appropriate `Err` variant.
- [ ] On failure, no state changes are observable (atomicity).
- [ ] On success, all additions are inserted, all removals are marked spent, chain tip is updated.

---

## Implementation Notes

- **Primary file:** `src/coin_store.rs` (or `src/block_application.rs`)
- **Dependencies:** BLK-002 through BLK-010 (apply_block orchestrates the full pipeline)
- **Edge cases:**
  - Empty `additions` and `removals` (e.g., a block with only coinbase rewards) is valid.
  - Genesis block (height 0) has special rules for parent hash (BLK-003) and coinbase coins (BLK-004).
  - Rollback after partial Phase 2 failure must restore all state.
